# Authentification | V-TOOLS API

L'authentification est cruciale dans une intégration web puisqu'elle permet a chaque utilisateur de faire des requêtes de manière indépendante et de coordonner les données.

Il existe deux manières d'authentification :

Authentification classiqueAuthentification Discord

L'Authentification classique est la connexion la plus rudimentaire de toutes, on a besoin d'un email et d'un mot de passe.

Pour faire fonctionner cette authentification, vous aurez besoin de souscrire un service de mail quelconque ainsi que de fournir certains liens de redirection.

**Email Service**

*   Email Address
    
*   Email Address Password
    
*   Email Address Host
    
*   Email Address Port
    
*   Email Variables
    

**Login Service**

*   Validate Account Redirect Uri
    
*   Reset Password Redirect Uri
    

L'Authentification Discord comme son nom indique utilise Discord OAuth2 pour authentifier les utilisateurs. Un compte Discord est donc requis.

Pour activer ce service vous devez avoir accès au panel Discord Developer.

Discord Service

*   Extern Container Id
    
*   Discord Api Client Id
    
*   Discord Api Client Secret
    

Les deux méthodes d'authentifications sont indépendantes l'une de l'autre mais pour une intégration web l'une d'entre elles doit être forcement choisie.

# RESTful API v3 | V-TOOLS API

Ceci marque le début de la troisième itération de la RESTful API de V-TOOLS.

Cette nouvelle version a pour but d'être extrêmement robuste et capable de supporter de belles charges serveur a elle seule.

Elle se découple en pleins de petits _microservice_ qu'elle desservira en tant que pont entre le client et le serveur ; avec une telle approche ca nous permet de réduire considérablement la charge serveur bien qu'elle contienne évidemment sa propre logique et abstraction.

L'objectif de cette troisième itération est aussi d'être le plus confidentielle possible quant a ces échanges et d'éviter de renvoyer des données inutiles.

Cette nouvelle API reste tout de même en développement bien qu'elle soit stable et utilisable ; des baisses de rendement par moment sont a déplorer et on travaille activement pour y remédier.

#### 

[](#format-des-requetes)

Format des requêtes

Si aucune spécification, l'ensemble des requêtes doivent s'effectuer de manière suivante.

Toutes les requetes commenceront par https://custom.v-tools.fr/v3/, et devront contenir les en-tetes (headers) suivantes :

*   `X-Server-Key: 45018c47-eb02-45...` qui permet d'identifier le serveur de provenance de la requête.
    
*   `Authorization: Bearer eyJhbGciOiJIUzI1NiIsI...` qui selon le cas identifie d'une manière ou d'une autre l'auteur de la requête.
    

En ce qui concerne le `X-Server-Key`, un administrateur vous fournira votre identifiant.

# Abonnements | V-TOOLS API

Les abonnements sont une fonctionnalité optionnelle mais intéressante qui vous permet de moduler vos accès selon vos différentes offres.

Veuillez demander à un administrateur de vous activer la fonctionnalité en mentionnant quel support vous utilisez pour créer, gérer et administrer vos abonnements.

Les différents supports actuellement disponibles sont :

DiscordStripe

Le support Discord est une bonne alternative lorsque les abonnements sont administrés via Whop. C'est le support le moins sophistiqué mais tout de même efficace.

Il se base tout simplement sur les rôles de l'utilisateur au sein du serveur Discord pour lui attribuer les différents accès.

Vous devrez avoir implémenté l'[Authentification Discord](/restful-api/restful-api-v3/authentification#authentification-discord)
 au préalable.

Un support Whop indépendant arrivera prochainement si nécessaire.

Le support Stripe est la meilleure option lorsque les abonnements sont administrés via Stripe. Ce support est assez complet et permet la gestion des stocks ainsi que la mise en place modulaire d'abonnements avec des périodes et jours d'essai au besoin.

Pour faire fonctionner ce service, les informations suivantes seront nécessaires :

**Stripe Service**

*   Stripe Api Key
    
*   Stripe Api Webhook Secret
    
*   Stripe Purchase Redirect Uri
    
*   Stripe Customer Portal Uri
    

#### 

[](#initialisation)

Initialisation

La création d'un endpoint webhook est nécessaire pour compléter la mise en place du service Stripe. Le lien de l'endpoint sera `https://custom.v-tools.fr/v3/store/subscription/w?server_key={{X_SERVER_KEY}}` et il faudra activer les événements suivants :

*   checkout.session.completed
    
*   customer.subscription.deleted
    
*   customer.subscription.updated
    
*   invoice.payment\_failed
    
*   invoice.payment\_succeeded

L'Authentification Discord tout comme la Classique se présente en plusieurs étapes. Cependant elle ne nécessite d'aucune vérification supplémentaire.

[](#lien-discord-oauth2)

🔗 Lien Discord OAuth2


----------------------------------------------------

Discord utilise l'OAuth2 pour nous autoriser l'accès aux informations relatives a un compte Discord. L'utilisateur doit donc accepter, avant de nous permettre d'accéder a ses données, d'où les étapes supplémentaires.

> Exemple lien Discord OAuth2 : [https://discord.com/oauth2/authorize?client\_id=1050044067234463784&response\_type=code&redirect\_uri=https%3A%2F%2Fapp.v-tools.fr%2Flogin&scope=identify+email](https://discord.com/oauth2/authorize?client_id=1050044067234463784&response_type=code&redirect_uri=https%3A%2F%2Fapp.v-tools.fr%2Flogin&scope=identify+email)
> .

Notez que ce lien contient les informations concernant le lien ou vous serez redirige une fois la validation complète.

Ce lien doit être généré depuis votre espace développeur Discord en cochant les scopes `identity` & `email`.

[](#se-connecter-a-un-compte-discord)

🎯 Se connecter a un compte Discord


------------------------------------------------------------------------------

Une fois l'OAuth2 valide, notez que la redirection contient un _query param_ `code`.

Ce code est la preuve que l'authentification s'est parfaitement déroule et celui qui permettra de récupérer les informations relatives a l'utilisateur.

On peut donc maintenant procéder a la connexion du compte Discord :

Une fois connecté, les tokens JWT fournis nous serviront d'_Authorization_ pour les prochaines requêtes, donc a conserver!

Le _query param_ `redirect` correspond au lien de redirection contenu dans le lien Discord OAuth2.

* * *

N'oubliez pas de spécifier le `X-Server-Key` dans l'en-tête (header) de la requête.

[](#lier-son-compte-discord)

🥨 Lier son compte Discord


------------------------------------------------------------

Une fois l'[OAuth2 valide](/restful-api/restful-api-v3/authentification/authentification-discord#lien-discord-oauth2)

[](#tokens-jwt)

⛳ Tokens JWT


---------------------------------

Les tokens JWT sont au cœur de l'API et servent de clés d'authentification pour tous les utilisateurs. Si l'intégration a été réalisée avec succès, vous devriez désormais être en mesure de générer vos propres tokens. Voici comment les utiliser.

#### 

[](#access-token)

Access Token

L'_access token_, ou token d'accès, est le plus important, et l'ensemble de votre future intégration repose dessus. Il est différent à chaque nouvelle connexion et regroupe toutes les informations nécessaires à la bonne identification de l'émetteur.

Sa durée de vie est relativement courte, **environ trois jours (au moment de la rédaction)**.

#### 

[](#refresh-token)

Refresh Token

Le _refresh token_ (le terme en français sonne moins bien) est plus trivial mais reste extrêmement intéressant. Comme son nom l'indique, il permet de rafraîchir le token d'accès pour prolonger sa durée de vie sans avoir à reconnecter l'utilisateur avec ses identifiants.

Une autre particularité est qu'il ne change pas forcément entre les connexions, mais seulement lorsqu'il rafraîchit un autre token.

Sa durée de vie est beaucoup plus longue, **environ trente jours (au moment de la rédaction)**.

Le seul inconvénient est qu'il n'existe pas de token pour rafraîchir le _refresh token_. À son expiration, il faudra donc reconnecter l'utilisateur.

Pour rafraichir les tokens a l'aide du _refresh token_ :

Remplacer l'`Authorization` par le _refresh-token_ uniquement pour cette fois dans l'en-tête (header) de la requête.

Il me semble évident de mentionner qu'à expiration, les tokens ne seront plus valides et l'API ne les acceptera plus. D'où tout l'intérêt de pouvoir les rafraîchir.

[](#profil)

🦚 Profil


--------------------------

La récupération du profil de l'utilisateur est une étape cruciale qui marque la fin de l'Authentification.

Le profil de l'utilisateur regroupe l'ensemble des données nécessaires à la bonne initialisation de l'application. Il contient notamment les permissions attachées à l'utilisateur (qui lui donneront accès par la suite à certaines requêtes), ainsi que l'information indiquant si l'utilisateur doit avoir accès à l'application (utile lorsqu'on a des abonnements).

L'API utilise la technologie Redis pour conserver en mémoire les dernières réponses et permettre une récupération et réutilisation relativement rapides des données.

L'ensemble des routes (_endpoints_) qui renvoient des données volumineuses intègre maintenant cette technologie. L'intégration est définitive et non optionnelle, il est donc impossible de choisir de ne pas utiliser la réutilisation du cache ; aucune option ne le permet pour l'instant.

La technologie Redis est également implémentée pour l'ensemble des websockets, donc tous les messages reçus via les websockets sont disponibles pour une réutilisation ultérieure selon vos besoins.

Sauf indication contraire, le cache a une durée de 24 heures. L'accès à la ressource rétablit son expiration.

Les filtres permettent ainsi de créer des chemins entre vous et le flux principal d'articles.

[](#recuperer-ses-filtres)

🥇 Récupérer ses filtres


--------------------------------------------------------

La récupération des filtres utilise la pagination. La limite maximale du nombre de filtres à récupérer à la fois est de 100. Vous devrez donc utiliser le paramètre `after` pour vous déplacer dans la liste.

[](#creer-un-nouveau-filtre)

🪩 Créer un nouveau filtre


------------------------------------------------------------

La création d'un nouveau filtre est flexible, car elle vous permet soit de renseigner champ par champ, soit d'utiliser directement un lien Vinted contenant toutes les informations basiques relatives au filtre. Pour procéder à la création :

Le champ `website_metadata.boosted` se réfère a l'[Autocop](/restful-api/restful-api-v3/vinted/transactions#achat-automatique)
. Vous devez posséder les permissions nécessaires avant de pouvoir cocher cette case.

[](#modifier-un-filtre-existant)

🧸 Modifier un filtre existant


--------------------------------------------------------------------

La modification d'un filtre préexistant est extrêmement similaire à la création d'un tout nouveau. La seule différence réside dans le fait de devoir renseigner l'identifiant du filtre que vous désirez modifier. Pour procéder à la modification :

[](#supprimer-un-filtre-existant)

----------------------------------------------------------------------

La suppression d'un filtre nécessite uniquement l'identifiant du ou des filtres concernés. Pour procéder à la suppression :

🌱 Se connecter à un compte Vinted


----------------------------------------------------------------------------

Tout comme lorsqu'on lie un compte Discord, lorsqu'on crée un nouveau compte Vinted, celui-ci nous est directement attribué. Pour procéder à sa création :

Depuis que Vinted a renforcé sa politique anti-bot, la connexion automatique est devenue plus complexe avec l'arrivée de l'A2F obligatoire.

Les détenteurs des comptes Vinted vont donc recevoir un code de confirmation lors de l'exécution.

Des réponses partielles (HTTP 206) seront donc retournées pour indiquer la bonne réception et compréhension de la demande, mais avec besoin de validation supplémentaire.

#### 

[](#valider-la2f)

Valider l'A2F

Le compte Vinted est maintenant associé au compte et des modifications peuvent y être apportées.

# Erreurs | V-TOOLS API

La structure d'une erreur est générique et statique sur l'ensemble de l'API, immuable. Elle contiendra constamment un code d'erreur ainsi que son message d'erreur associé.

Voici un exemple des différentes erreurs que vous pourriez rencontrer.

La plupart des erreurs se basent sur le contexte de la requête, donc il est important de savoir ce que vous faites a tout moment.

#### 

[](#invalid-operation)

Invalid Operation

Généralement lorsque vous rencontrez _Invalid Operation_ c'est un simple manque de documentation a propos de l'erreur donc réclamez le changement auprès d'un administrateur.

Copy

    {
        "data": "Invalid Operation.",
        "code": 1
    }

#### 

[](#illegal-access)

Illegal Access

_Illegal Access_ decrit une tentative d'accès erronée a la ressource soit par un manquement des bons credentials ou bien simplement une ressource qui ne vous est pas propre.

Cette erreur est généralement trouve dans l'[Authentification](/restful-api/restful-api-v3/authentification)
.

Copy

    {
        "data": "Illegal Access.",
        "code": 1
    }

#### 

[](#invalid-account)

Invalid Account

_Invalid Account_ decrit une tentative d'accès a une ressource appartenant a un compte Vinted étranger.

Cette erreur est généralement aperçu lors de l'intégration de [Vinted](/restful-api/restful-api-v3/vinted)
.

Copy

    {
        "data": "Invalid Account.",
        "code": 1
    }

#### 

[](#not-authorized-not-allowed)

Not Authorized / Not Allowed

_Not Authorized_ décrit une tentative d'accès a la ressource avec une `Authorization` invalide.

Copy

    {
        "data": "Not Authorized.",
        "code": 10
    }

#### 

[](#invalid-body-invalid-response)

Invalid Body / Invalid Response

_Invalid Body_ décrit une corps de requête (body) ou de réponse non conforme, c'est souvent un problème cause cote client.

Copy

    {
        "data": "Invalid Body.",
        "code": 20
    }

Copy

    {
        "data": "Invalid Response.",
        "code": 1
    }

#### 

[](#not-subscribed)

Not Subscribed

_Not Subscribed_ décrit une tentative d'accès a la ressource sans avoir souscrit un abonnement.

Copy

    {
        "data": "Not Subscribed.",
        "code": 11
    }

#### 

[](#subscription-payment-failed)

Subscription Payment Failed

_Subscription Payment Failed_ décrit un échec de paiement donc un interdiction d'accès a la ressource.

Copy

    {
        "data": "Subscription Payment Failed.",
        "code": 12
    }

#### 

[](#server-do-not-meet-requirements)

Server Do Not Meet Requirements

_Server Do Not Meet Requirements_ décrit un manquement dans la configuration du serveur, veuillez contacter un administrateur a ce moment la.

Copy

    {
        "data": "Server Do Not Meet Requirements (xxx).",
        "code": 70
    }

#### 

[](#server-has-been-suspended)

Server Has Been Suspended

_Server Has Been Suspended_ décrit une suspension forcée des activités du serveur, souvent lié a un manque de paiement de la part de l'abonné (gérant du serveur).

Copy

    {
        "data": "Server Has Been Suspended.",
        "code": 71
    }

#### 

[](#not-registered-issuer)

Not Registered Issuer

_Not Registered Issuer_ décrit une tentative d'accès a la ressource avec un `X-Server-Key` invalide.

Copy

    {
        "data": "Not Registered Issuer.",
        "code": 72
    }

* * *

Cette page recapitule uniquement les erreurs les plus communes et generales, d'autres erreurs bien plus specifiques peuvent tout de meme survenir.

# Évènement | V-TOOLS API

La strucutre d'un evenement emis par le websocket est statique et ne sera soumise a aucun changement, toutefois le contenu des evenements est independant et eux pourront subir des modifications, donc une particuliere est requise.

Voici une liste des évènements actuellement disponibles.

#### 

[](#buy_attempt)

buy\_attempt

L'évènement _buy\_attempt_ est émis lorsqu'un utilisateur lance le paiement d'un article (uniquement a la dernière étape du paiement).

Copy

    {
      "content_type": "event",
      "data": {
        "target": "**",
        "event": "buy_attempt",
        "data": {
          "who": "65a966017da83ab7cfdcaf95",
          "item_id": 4699404456,
          "boosted": false
        },
        "created_at": "2024-01-01T00:00:00.000000000Z"
      }
    }

#### 

[](#buy_failed)

buy\_failed

L'évènement _buy\_failed_ est émis lorsqu'un utilisateur lance le paiement d'un article mais échoue.

Copy

    {
      "content_type": "event",
      "data": {
        "target": "65a966017da83ab7cfdcaf95",
        "event": "buy_failed",
        "data": {
          "who": "65a966017da83ab7cfdcaf95",
          "item_id": 4699404456,
          "error": "Le paiement a échoué (Generic Error Y01). Si le problème persiste, contacte-nous."
        },
        "created_at": "2024-01-01T00:00:00.000000000Z"
      }
    }

#### 

[](#buy_completed)

buy\_completed

L'évènement _buy\_completed_ est émis lorsqu'un utilisateur lance le paiement d'un article avec succès.

Copy

    {
      "content_type": "event",
      "data": {
        "target": "**",
        "event": "buy_completed",
        "data": {
          "who": "65a966017da83ab7cfdcaf95",
          "item_id": 4699404456,
          "3ds": "https://payment.vinted.fr/confirm",
          "scope": "autocop", // present uniquement lors d'autocop
          "boosted": true // false -> classique
        },
        "created_at": "2024-01-01T00:00:00.000000000Z"
      }
    }

#### 

[](#sniper_toggle)

sniper\_toggle

L'évènement _sniper\_toggle_ est émis lors d'un changement d'état de l'autocop. Il simule si l'activation/désactivation de l'autocop en temps réel.

Copy

    {
      "content_type": "event",
      "data": {
        "target": "65a966017da83ab7cfdcaf95",
        "event": "sniper_toggle",
        "data": {
          "status": true // false -> l'autocop est maintenant inactif
        },
        "created_at": "2024-01-01T00:00:00.000000000Z"
      }
    }
