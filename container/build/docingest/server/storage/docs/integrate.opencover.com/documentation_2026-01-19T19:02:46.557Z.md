# Introduction | Insured Metavaults | OpenCover

![Page cover](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/~gitbook/image?url=https%3A%2F%2F91775999-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FHgV6a19J58nGSdTzSPIq%252Fuploads%252FwiFvY0qZtHOJLDARdmqY%252Fopencover-header.avif%3Falt%3Dmedia%26token%3D910c9594-828d-41c6-a418-86769b66de55&width=1248&dpr=4&quality=100&sign=5ca2e50a&sv=2)

OpenCover and Nexus Mutual are joining forces to deliver insured DeFi yield at institutional scale for the first time. Building on the industry’s longest-standing experience in onchain risk transfer and the deepest network of onchain underwriting capital, Insured Metavaults introduce a new primitive that addresses the hardest challenges in onchain insurance, from premium streaming to renewal abstraction. This enables virtually any DeFi vault to become insurable without modifying the associated protocol or primary point of deposit.

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults#user-journey)

User journey

Metavaults work by wrapping underlying vault shares and binding their market value to a publicly auditable insurance policy backed by onchain capital. The user journey can be summarised as follows:

1.  Yield-bearing vault shares (e.g. [Morpho Smokehouse USDCarrow-up-right](https://app.morpho.org/ethereum/vault/0xBEeFFF209270748ddd194831b3fa287a5386f5bC/smokehouse-usdc)
    ) are deposited to an associated Metavault.
    
2.  Insurance coverage begins once the deposit request settles in the Metavault (typically within 2 Ethereum blocks, about 30 seconds). There is no upfront cost to the user for coverage.
    
3.  Insurance premiums are streamed from the yield accrued in the wrapped vault shares, eliminating the need to manage renewals. Coverage rolls automatically as long as shares remain deposited.
    
4.  Upon withdrawal of the vault shares from the Metavault, coverage ends.
    

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults#compatibility)

Compatibility

Insured Metavaults effectively extend protocols by enabling users to add coverage to their positions on demand and can be chained with native deposit and withdrawal actions.

Insured Metavaults are compatible with the [ERC‑7540arrow-up-right](https://eips.ethereum.org/EIPS/eip-7540)
 asynchronous vault standard, enabling smooth integration with all common vault interfaces.

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults#compliant-and-non-custodial)

Compliant and non‑custodial

Metavaults can be access-controlled and are non‑custodial at all times. Deposits can be subject to integrator whitelists while withdrawals are unconditional, guaranteed and enforced onchain.

# Claims | Insured Metavaults | OpenCover

Claims are managed by the Metavault owner (typically OpenCover) using the associated Metavault policy, or policies, bound on Nexus Mutual, with payouts disbursed to the active Metavault depositors (C-token holders).

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/insurance/claims#claim-process)

Claim process

After a covered incident occurs, a cool-off period is observed to confirm the loss is final and the associated amount. The duration of the cool-off period is specified in the Metavault policy schedule (typically 14 days).

The vault owner then submits a claim to Nexus Mutual for assessment and the claim is evaluated by [3-5 specialist claim assessorsarrow-up-right](https://forum.nexusmutual.io/t/nmpip-261-reform-the-claims-mechanism-to-build-trust-upgrade-governance-to-formally-use-an-optimistic-model/1790)
. Payouts are made from Nexus Mutual to a designated multisig controlled by the Metavault owner, used to disburse funds to the individual active Metavault depositors affected by the loss (C-token holders). The entire process typically takes five calendar days from claim submission.

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/insurance/claims#recovery)

Recovery

In line with traditional insurance practices and to discourage fraud, once a claim is paid out, rights over any recoveries associated with the fund loss event, such as a protocol attacker refunding victims, are assigned to Nexus Mutual.

The policy schedule, detailing the exact covered risks and cover terms, is stored on IPFS and immutably linked onchain as part of the Metavault deployment. # Metavault lifecycle | Insured Metavaults | OpenCover

![](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/~gitbook/image?url=https%3A%2F%2F91775999-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FHgV6a19J58nGSdTzSPIq%252Fuploads%252FEmJlh8wB5IOGRp6L0dCG%252FInsured%2520Metavault%2520Lifecycle%2520%25235.drawio.svg%3Falt%3Dmedia%26token%3D802c47d4-0fe9-4c81-b95d-72896e7f13d4&width=768&dpr=4&quality=100&sign=8d9d5041&sv=2)

Metavault example for Smokehouse USDC on Morpho

#### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/metavault-lifecycle#deposit)

Deposit

1.  User deposits an asset (e.g. **USDC**) into a supported vault (e.g. [Morpho Smokehouse USDCarrow-up-right](https://app.morpho.org/ethereum/vault/0xBEeFFF209270748ddd194831b3fa287a5386f5bC/smokehouse-usdc)
    ) to mint and receive vault shares (e.g. **bbqUSDC**).
    
2.  User submits a _deposit request_ to a Metavault configured to cover the relevant vault shares (e.g. **bbqUSDC**)
    
    1.  The request is recorded and the vault shares are held by the Metavault pending settlement.
        
    2.  This request can be _cancelled_ by the user at any time before settlement to withdraw the vault shares pending settlement.
        
    
3.  The deposit request is typically settled within 2 Ethereum blocks (<30 seconds). On settlement, insurance coverage becomes active and the Metavault issues covered shares (**C-bbqUSDC**) to the user.
    
4.  Yield continues accruing in the underlying vault. The insurance premium is streamed from accrued yield for as long as cover remains active.
    

#### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/metavault-lifecycle#withdrawal)

Withdrawal

1.  User submits a _redemption request_ by providing covered shares (**C-bbqUSDC**) to the Metavault.
    
    1.  The request is recorded, shares are locked pending settlement and continue to be insured.
        
    2.  The user can _cancel_ the redemption request any time before settlement.
        
    
2.  On redemption settlement, coverage ceases for the redeemed vault shares (**bbqUSDC**), which become available to withdraw (ERC-7540 native flow) or can be pushed to the user's wallet depending on desired user experience (OpenCover extended ERC-7540 standard).
    
3.  In the unlikely event the redemption request does not settle, the onchain exit guarantee allows the user to claim their vault shares (**bbqUSDC**) 24 hours from request creation.

# Monitoring | Insured Metavaults | OpenCover

Insured Metavaults implement continuous monitoring and a circuit breaker mechanism. The Metavault's **Guardian** role serves as the last line of defense, with the authority to pause vault operations in the unlikely event that a vulnerability goes undetected during audit.

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/security/monitoring#what-we-monitor)

What we monitor


-------------------------------------------------------------------------------------------------------------------------------------------

🏦 Metavault health

⌛ Request queues

🤝 Governance changes

The system tracks core vault metrics and invariants in real-time:

*   Sudden changes in Metavault assets (TVL);
    
*   Share price movements and exchange rate stability;
    
*   Premium streaming activity and collection rates;
    
*   Pending deposit and redemption queue sizes.
    

Anomalies trigger automatic alerts when thresholds are breached, such as TVL drops exceeding 20% or share price movements beyond expected volatility bands.

Asynchronous deposit and redemption requests are monitored for:

*   Unusual surge patterns that may indicate malicious attempts to grief the vault;
    
*   Settlement delays beyond business-as-usual operations.
    

The system maintains liveness guarantees, ensuring pending requests are processed within defined timeframes.

All privileged operations generate immediate alerts:

*   Role assignments and revocations;
    
*   Premium rate adjustments;
    
*   Pause and unpause events.
    

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/security/monitoring#circuit-breaker)

Circuit breaker

The **Guardian** can trigger an emergency pause when monitoring detects:

*   Potential exploit patterns or abnormal contract interactions;
    
*   Unauthorised administrative changes;
    
*   Critical invariant violations;
    
*   Settlement mechanism failures.
    

Once paused, operations resume only after the issue is resolved and validated through simulation testing.

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/security/monitoring#operations)

Operations


---------------------------------------------------------------------------------------------------------------------------------

The vault maintains strict separation of concerns: alongside the Guardian, other roles include the **Keeper** (settlement operations), **Manager** (premium configuration) and **Owner** (upgrades). The Guardian is exclusively permissioned to pause and unpause the vault, ensuring independent risk assessment.

The **Guardian** role is held by a secure multisig with defined procedures:

*   24/7 on-call rotation with hardware wallet signers;
    
*   Pre-approved emergency response playbooks and drill exercises to maintain readiness;
    
*   Post-incident reviews to improve response protocols.

# Getting started | Insured Metavaults | OpenCover

Insured Metavaults provide onchain integration on Ethereum (more chains coming soon), enabling integrators to offer native covered yield by depositing yield‑bearing vault shares. Coverage starts automatically at settlement, with premiums streaming from accrued yield.

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#integration-overview)

Integration overview


-------------------------------------------------------------------------------------------------------------------------------------------------------------

circle-info

The contract supports ERC‑7540's two-step asynchronous lifecycle for deposits and withdrawals, as well as extended single-transaction deposit/withdrawal flows, both preserving non‑custodial ownership. In both cases, coverage activates at deposit settlement and detaches at redemption settlement.

As an example case, we cover Steakhouse Financial's [Smokehouse USDC Morpho vaultarrow-up-right](https://app.morpho.org/ethereum/vault/0xBEeFFF209270748ddd194831b3fa287a5386f5bC/smokehouse-usdc)
 and assume the user already holds **bbqUSDC** shares that they wish to insure through the Metavault.

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#deposit-flow)

Deposit flow

To start earning insured yield, the user deposits vault shares from the yield-vault (**bbqUSDC** in case of Morpho Smokehouse USDC) into the Insured Metavault.

1

#### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#submit-a-deposit-request)

Submit a deposit request

*   Create a transaction calling the `requestDeposit(asset, controller, owner)` function on the Metavault contract to create a deposit request.
    
*   Use `pendingDepositRequest(requestId, controller)` to fetch assets pending coverage.
    
*   Requests can be cancelled any time before settlement.
    

Copy

    // Request to deposit 10,000 bbqUSDC.
    metavault.requestDeposit(10_000e18, msg.sender, msg.sender);
    
    // Fetch assets pending coverage.
    uint256 assetsPending = metavault.pendingDepositRequest(0, msg.sender);

2

#### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#settlement-and-coverage-activation)

Settlement & coverage activation

*   Requests are batch‑settled typically within 2 Ethereum blocks (<30 seconds).
    
*   On settlement, the deposit either becomes claimable (users can mint covered Metavault shares, e.g. **C-bbqUSDC**) or Metavault shares are directly pushed to the user wallet (integrator preference). In either case, coverage activates for the settled amount.
    
*   Coverage auto-renews while shares remain in the Metavault.
    

Copy

    // Get covered assets for which Metavault shares can be minted.
    uint256 assetsClaimable = metavault.claimableDepositRequest(0, msg.sender);

3

#### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#mint-covered-metavault-shares)

Mint covered Metavault shares

*   Create a transaction calling either the `deposit(assets, receiver, controller)` or `mint(shares, receiver, controller)` functions of the Metavault following [ERC-4626 semanticsarrow-up-right](https://ethereum.org/developers/docs/standards/tokens/erc-4626/#deposit)
    .
    
*   The user receives **C-bbqUSDC** shares (covered **bbqUSDC**), a standard ERC-20 token representing their insured position.
    

Copy

    // Mint C-bbqUSDC Metavault shares for the bbqUSDC deposit.
    metavault.deposit(assetsClaimable, msg.sender, msg.sender);

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#withdrawal-flow)

Withdrawal flow

When leaving the Metavault, the user redeems **C-bbqUSDC** for **bbqUSDC**.

1

#### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#submit-a-redemption-request)

**Submit a redemption request**

*   Call `requestRedeem(shares, controller, owner)` with shares specified in covered Metavault shares (e.g. **C-bbqUSDC**).
    
*   The Metavault records the request and locks the shares.
    
*   Use `pendingRedeemRequest(requestId, controller)` to fetch shares pending redemption.
    
*   Requests can be cancelled any time before settlement.
    

Copy

    // Redeem all covered shares (C-bbqUSDC) for the deposited asset (bbqUSDC).
    uint256 shares = metavault.balanceOf(msg.sender);
    uint256 reqId = metavault.requestRedeem(shares, msg.sender, msg.sender);

2

#### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#settlement-and-coverage-end)

Settlement & coverage end

*   Pending withdrawal requests settle in FIFO order.
    
*   On settlement, coverage ends for the redeemed portion and the corresponding **bbqUSDC** becomes claimable or can be directly pushed to the user wallet (integrator preference).
    

Copy

    // After settlement, the redeemed shares become claimable.
    uint256 sharesClaimable = metavault.claimableRedeemRequest(reqId, msg.sender);

*   Withdrawal requests typically settle within 2 Ethereum blocks (<30 seconds), pending incoming deposits and underlying policy expiry.
    
*   If not settled within 24 hours, the onchain exit guarantee enables the user to withdraw **bbqUSDC** permissionlessly.
    

Copy

    // After 28 days, redemption requests can be user-settled.
    metavault.settleMaturedRedemption(reqId);

3

#### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#receive-yield-vault-shares)

**Receive yield-vault shares**

*   Call either `redeem(shares, receiver, controller)` or `withdraw(assets, receiver, controller)` to burn **C-bbqUSDC** and receive **bbqUSDC**.
    

Copy

    metavault.redeem(sharesClaimable, msg.sender, msg.sender);

*   _Optional:_ Redeem **bbqUSDC** for **USDC** directly at the underlying Morpho vault.
    

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/insured-metavaults/integration/getting-started#full-integration-example)

Full integration example


---------------------------------------------------------------------------------------------------------------------------------------------------------------------

Copy

    ICoveredMetavault metavault = ICoveredMetavault(0x...); // Address TBC.
    IERC4626 bbqUSDC = IERC4626(0xBEEF01735c132Ada46AA9aA4c54623cAA92A64CB);
    
    // 1. Approve the Metavault.
    bbqUSDC.approve(address(metavault), 10_000e18);
    
    // 2. Request to deposit 10,000 bbqUSDC.
    metavault.requestDeposit(10_000e18, msg.sender, msg.sender);
    
    // 3. Wait for settlement (typically within 2 blocks) then check claimable assets.
    uint256 assetsClaimable = metavault.claimableDepositRequest(0, msg.sender);
    
    // 4. Mint covered shares (C-bbqUSDC); 
    //    or, if preferred, shares can be pushed as part of settlement.
    metavault.deposit(assetsClaimable, msg.sender, msg.sender);
    
    // ... Time passes, user earns insured yield ...
    
    // 5. Get covered metavault share balance of user (C-bbqUSDC).
    uint256 sharesCovered = metavault.balanceOf(msg.sender);
    
    // 6. Request redemption of covered shares.
    uint256 reqId = metavault.requestRedeem(sharesCovered, msg.sender, msg.sender);
    
    // 7. Wait for settlement then check claimable shares.
    uint256 sharesClaimable = metavault.claimableRedeemRequest(reqId, msg.sender);
    
    // 8. Redeem metavault shares (C-bbqUSDC) for yield-vault shares (bbqUSDC).
    //    or, if preferred, yield-vault shares can be pushed as part of settlement.
    metavault.redeem(sharesClaimable, msg.sender, msg.sender);

# Getting started | OpenCover

A native integration enables you to embed the entire cover purchase flow directly within your platform's interface. Users complete all steps, from getting a quote to purchasing coverage within your platform.

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/getting-started#integration-overview)

Integration overview


-------------------------------------------------------------------------------------------------------------------------------------------------

circle-info

Native integration is supported on the following chains: **Ethereum**, **Base**, **Arbitrum**, **Optimism**, **Polygon**.

1

[**Get a quote via API**](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/integration-guide/get-a-quote)

*   Request premium options for cover on a specific [product](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/resources/products)
     (protocol, asset or ecosystem) given an amount and duration using the OpenCover API.
    
*   The API response will include cryptographically signed quotes in all supported currencies (e.g. USDC, cbBTC, ETH). These signed quotes are required for the onchain submission step.
    

2

[**Submit selected quote onchain**](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/integration-guide/submit-a-quote)

**Approve token transfer (if applicable)**

*   If the premium is to be paid using an ERC20 token (like USDC or cbBTC), the user must first execute an approve transaction. This grants the OpenCover Quote contract permission to collect the premium amount from their wallet.
    

**Submit the quote:**

*   The user submits the signed quote to the OpenCover Quote contract. This transaction executes the purchase and transfers the approved ERC20 premium (if applicable).
    
*   See [an example transactionarrow-up-right](https://basescan.org/tx/0xe90b7fd3298005d37719c57c7b969b2ede7cac4c8e99a7366946b53aef2c6eb7)
     on Base.
    
*   Submitted quotes are fulfilled by OpenCover within 24 hours. For more details, please see our [Quote lifecycle](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/resources/quote-lifecycle)
     page.
    

3

[**Track coverage status**](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/integration-guide/track-coverage)

Show users their cover purchase history and active coverage in real-time via OpenCover's easy-to-use API.

4

[**Enable cover notifications via API**](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/integration-guide/subscribe-to-notifications)
 **(optional)**

Offer users the ability to receive email notifications when their cover becomes active or is about to expire.

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/linkout-integration/integration-parameters#query-parameters)

Query parameters

*   `cover`
    
    *   Product to cover.
        
    *   **Allowed values:** [See available products](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/resources/products)
        .
        
    *   **Example:** [https://opencover.com/app/?**cover=97**arrow-up-right](https://opencover.com/app/?cover=97)
        
    
*   `amount`
    
    *   Amount of protection. Must be used with a valid `currency`.
        
    *   **Allowed values:** Any positive amount.
        
    *   **Example:** [https://opencover.com/app/?**amount=1000**&currency=USDCarrow-up-right](https://opencover.com/app/?amount=1000&currency=USDC)
        
    
*   `currency` _(default: USDC)_
    
    *   Currency for the protection amount.
        
    *   **Allowed values**: `USDC`, `ETH`, `cbBTC`
        
    *   **Example:** [https://opencover.com/app/?**currency=cbBTC**arrow-up-right](https://opencover.com/app/?currency=cbBTC)
        
    
*   `duration` _(default: 30)_
    
    *   Duration of the coverage in days.
        
    *   **Allowed values:** Any number from 30 to 180.
        
    *   **Example:** [https://opencover.com/app/?**duration=60**arrow-up-right](https://opencover.com/app/?duration=60)
        
    
*   `coveredAddresses` _(default: not set)_
    
    *   List of wallet addresses to prefill as covered wallets.
        
    *   **Allowed values:** Up to 20 valid wallet addresses (EVM-compatible), comma-separated. [Contact usarrow-up-right](https://airtable.com/appm7CYnxTomwS4Kg/shrkpntNzvkMC4tJz)
         to cover larger custom portfolios.
        
    *   **Example:** [https://opencover.com/basepass/?**coveredAddresses=0x742d35Cc6634C0532925a3b844Bc454e4438f44e,0x53d284357ec70cE289D6D64134DfAc8E511c8a3D**arrow-up-right](https://opencover.com/basepass/?coveredAddresses=0x742d35Cc6634C0532925a3b844Bc454e4438f44e,0x53d284357ec70cE289D6D64134DfAc8E511c8a3D)
        
    *   **Note:** This parameter applies **only** to [Ecosystem products](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/resources/products#ecosystem-cover)
         that require covered wallets to be specified. It will have no effect if used on [Non-Ecosystem products](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/resources/products#protocol-cover)
         as all wallets are covered in other products.
        
    
*   `screen` _(default: main)_
    
    *   Auto-fetch a quote when set to `quote`. Requires `cover`, `amount`, and `currency` to be set.
        
    *   **Allowed values:** `main`, `quote`
        
    *   **Example:** [https://opencover.com/app/?cover=211&amount=1000&currency=USDC&**screen=quote**arrow-up-right](https://opencover.com/app/?cover=211&amount=1000&currency=USDC&screen=quote)
        
    
*   `invite` _(default: direct)_
    
    *   Referrer invite code unique to the integrator.
        
    *   **Allowed values:** [Contact usarrow-up-right](https://airtable.com/appm7CYnxTomwS4Kg/shrkpntNzvkMC4tJz)
         to receive yours.
        
    *   **Example:** [https://opencover.com/app/?**invite=my-custom-code**arrow-up-right](https://opencover.com/app/?invite=my-custom-code)
        
    
*   `integrator` _(default: 0 aka OpenCover)_
    
    *   Identifier for the integrator.
        
    *   **Allowed values:** Unique integrator ID. [Contact usarrow-up-right](https://airtable.com/appm7CYnxTomwS4Kg/shrkpntNzvkMC4tJz)
         to receive yours.
        
    *   **Example:** [https://opencover.com/app/?**integrator=1337**arrow-up-right](https://opencover.com/app/?integrator=1337)
        
    
*   `network` _(default: 8453 aka_ _**Base**__)_
    
    *   The chain the user will be prompted to connect.
        
    *   **Allowed values:**
        
        *   `8453` **Base**
            
        *   `10` **Optimism**
            
        *   `42161` **Arbitrum**
            
        *   `137` **Polygon**
            
        
    *   **Example:** [https://opencover.com/app/?**network=137**arrow-up-right](https://opencover.com/app/?network=137)

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/api-endpoints#understanding-quote-statuses)

Understanding quote statuses

The response includes quotes in various states, each identified by its `quoteStatus`:

*   `QUOTE_AWAITING_SETTLEMENT` The quote has been submitted awaiting settlement by OpenCover.
    
*   `COVER_ACTIVE` Coverage is currently in force.
    
*   `COVER_EXPIRED` The coverage period has ended.
    
*   `QUOTE_EXPIRED` The quote was not settled within 24 hours, refund is available.
    
*   `QUOTE_REFUNDED` The premium payment was returned to the payer.
    

For more details on the possible states please see the [Quote lifecycle](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/resources/quote-lifecycle)
 page.

These statuses refresh in real-time as actions occur onchain, allowing to monitor the coverage without needing to manually check each transaction.

Each returned quote contains details about the covered product, coverage amount, premium paid, and key timestamps including `coverExpiresAt` (when coverage ends). Note that `settledAt` and `coverExpiresAt` will be `null` for quotes that are awaiting settlement or refunded.

Check if a user is signed up for notifications

This [GET](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/api-endpoints#api-user-address-chainid)
 endpoint returns the user's email address registered for OpenCover notifications. If the `X-OC-Signature` HTTP header is not included in the request, the response returns an obfuscated version of the email which is helpful for visual confirmation without requiring user action.

In this case, the response includes the `X-OC-Sign` HTTP header, containing a Base64-encoded OpenCover consent message. The user signs this message and sends the resulting signature in the `X-OC-Signature` header to the same [GET](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/api-endpoints#api-user-address-chainid)
 or [POST](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/api-endpoints#api-user)
 endpoints.

Special HTTP response status codes `401` and `404` also return the `X-OC-Sign` header for signature generation (see the **Responses** section below).

# Subscribe to notifications | OpenCover

Allow users to subscribe to or manage their email notifications for OpenCover covers using the API.

Notifications inform users about important events related to their cover, such as upcoming expiry.

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/integration-guide/subscribe-to-notifications#overview)

Overview


------------------------------------------------------------------------------------------------------------------------------------------------------

1.  **Check status & get consent message:** query the API to check if the user is already subscribed and retrieve a consent message.
    
2.  **Generate signature:** Prompt the user to sign the consent message with their wallet to prove ownership.
    
3.  **Subscribe / update / view:** Use the generated signature to either:
    
    1.  Subscribe a new email address for notifications or update an existing one.
        
    2.  View the full registered email address (optional).

1.  Decode the Base64-encoded message from the `X-OC-Sign` header.
    
2.  Use a library like _wagmi_ to prompt the user to sign the decoded message string.
    

The resulting `signature` string is used in the subsequent API calls.

# Submit a quote | OpenCover

This step finalises the cover purchase by submitting the selected quote and its signature to the OpenCover _Quote_ smart contract on the relevant L2 network using the user's connected wallet.

Submitting a quote involves potentially two transactions:

1.  **ERC20 approval**: If paying with an ERC20 token, prepare and execute an approve transaction.
    
2.  **Quote submission**: Prepare and execute the cover purchase transaction.
    

See the page below for the deployed contract addresses and ABI below.

[file-signatureSmart contractschevron-right](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/resources/contracts)

### 

[hashtag](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/native-integration/integration-guide/submit-a-quote#approve-erc20-transfer-if-applicable)

Approve ERC20 transfer (if applicable)

If the payment is going to be made in an ERC20 token (USDC, USDT or cbBTC), the user needs to execute an approve transaction for spending the payment amount of that ERC20 token by the _Quote_ contract.

*   **Goal:** Call the `approve()` function on the ERC20 token contract with the appropriate amount.
    

The example below uses [_wagmi_arrow-up-right](https://wagmi.sh/core/getting-started)
 to approve purchasing a cover in USDC.

Copy

    import { useWriteContract, useReadContract } from 'wagmi';
    
    const quote = {}; // HTTP POST to https://opencover.com/api/quote, see "Get a quote"
    
    // If a user decides to pay in USDC, find the corresponding payment quote
    const paymentQuote = quote.quotes.find(q => q.premium.currency.symbol === 'USDC');
    
    // Base mainnet Quote contract address
    const contractAddress = '0xD68647555e5da198d50866334EEd647cbE3d1556';
    
    // Read asset metadata to get ERC20 contract address
    const { data: asset } = useReadContract({
        address: contractAddress,
        abi: QuoteAbi,
        functionName: 'assets',
        args: [quote.providerId, paymentQuote.premium.currency.assetId]
    });
        
    const { writeContract } = useWriteContract();
    
    // Approve OpenCover Quote contract to allow ERC20 payment
    writeContract({
        address: asset.assetAddress,
        abi: ERC20Abi,
        functionName: 'approve',
        args: [\
            contractAddress,\
            BigInt(paymentQuote.premium.value),\
        ],
    });

Response

The response includes a list of quotes and their current status in the `quoteStatus` field:

*   `QUOTE_AWAITING_SETTLEMENT` The quote has been submitted, awaiting settlement by OpenCover.
    
*   `COVER_ACTIVE` Coverage is active and protection is live.
    
*   `COVER_EXPIRED` The coverage period has ended.
    
*   `QUOTE_EXPIRED` The quote was not settled within 24 hours, refund is available.
    
*   `QUOTE_REFUNDED` The premium payment was returned to the payer.
    

For more details on the possible statuses please see the [Quote lifecycle](https://integrate.opencover.com/zeiNPSbmoJnSDblPJcJK/resources/quote-lifecycle)
 page.

The example API response below shows 3 quotes in various states:

1.  **Pending** 60-day **vfat** cover for 1,900 USDC
    
2.  **Active** 30-day **ExtraFi** cover for 27,500 USDC
    
3.  **Expired** 30-day **ExtraFi** cover for 47,000 USDC
