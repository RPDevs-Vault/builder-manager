# Introducción | Dart

Dart (originalmente llamado Dash) es un lenguaje de programación de **código** **abierto**, **desarrollado** por **Google** y presentado oficialmente en la conferencia _goto_; en Aarhus, Dinamarca el 10 de octubre de **2011**.

[_wikipedia_](https://en.m.wikipedia.org/wiki/Dart_(programming_language))
_: Dart es un lenguaje de programación de propósito general originariamente desarrollado por Google y después aprobado como estándar por la Ecma (ECMA-408). Dart se emplea para construir aplicaciones para móviles, servidores, webs y también para internet de las cosas (IoT). Dart es de código abierto y está bajo la licencia BSD._

#### 

[](https://ecm-pmdm-flutter.gitbook.io/dart#intenciones-iniciales)

Intenciones iniciales

*   Con Dart, google buscaba inicialmente ser una alternativa moderna a **JavaScript** para la programación web, querían incluirlo en Chrome (causando cierto nerviosismo entre los desarrolladores).
    
*   Finalmente, en 2015 se optó por transpilar (transcompile) Dart a JavaScript y actualmente cuenta con múltiples opciones de compilación.
    

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart#caracteristicas)

Características

Algunas de sus características son:

*   Dart es **multiparadigma**: imperativo, orientado a objetos, reflectivo, funcional y de herencia simple.
    
*   **Sintaxis** similar al lenguaje **C** que puede **transpilarse** a lenguaje _**JavaScript**_.
    
*   Soporta: interfaces, mixins, clases abstractas, genéricos, tipado estático, **tipado** **opcional**, y mucho más.
    
*   El código de Dart es _**single**_ _**threaded**_. La **concurrencia** se implementa con _**isolates**_ que son unidades de ejecución independientes pero que no comparten nunca estado. Los _isolates_ se comunican con el paso de mensajes.
    
*   Dart es **null-safe**: por defecto, las variables no pueden tener un valor _null_ a no ser que lo digamos explícitamente.
    

Dart és **multiplataforma**, tanto para dispositivos **móviles**, **web**, **escritorio** y dispositivos **embebidos**.

![](https://ecm-pmdm-flutter.gitbook.io/dart/~gitbook/image?url=https%3A%2F%2F1551469701-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FsGv3FSfnl3pEE6ftH9Ys%252Fuploads%252FV0n9m8k9AQX8bDfN3nLw%252Fimage.png%3Falt%3Dmedia%26token%3Dcaa7fc9d-2d1f-4476-b4b0-ba7502ee7354&width=768&dpr=4&quality=100&sign=3f6a9325&sv=2)

Además:

*   Utiliza un **compilador AOT (Ahead-of-Time):** Se compila todo el código entero a nativo antes de la ejecución.
    
*   Durante el desarrollo, permite realizar un **hot reload** que permite ver de forma **instantánea** los cambios realizados en la aplicación.
    
*   Independientemente de la plataforma o cómo se compile, la ejecución del código requiere un **runtime**: Utiliza una máquina virtual llamada **Dart VM** (gestión de memoria, garbage collector, gestión de isolates, etc)
    
*   Las **aplicaciones** de **Flutter** utilizan Dart como lenguaje de desarrollo.
    

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart#herramientas-para-programar-con-dart)

    ) y un editor de texto o un **IDE** con el **plugin** **Dart**:
    
    *   [IntelliJIDEA](https://www.jetbrains.com/help/idea/dart.html)
        
    *   [Visual Studio Code](https://code.visualstudio.com/)
        
    *   [Atom](https://atom.io/)

        , etc.
        
    
*   **Desde la Web:** Dart también proporciona una **interfaz** **web** para escribir y probar código Dart: [https://dartpad.dev/](https://dartpad.dev/)

Una **aplicación** **Flutter** se **ejecuta** de forma **nativa** en cada plataforma, gracias a la **compilación** Ahead Of Time (**AOT**).

![](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2F3645852152-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FTJQsPEhsPLjjuhB6XVoa%252Fuploads%252FzEvLI5BcEfT6TX0PFsyy%252Fimage.png%3Falt%3Dmedia%26token%3D3249c206-516a-44ce-8236-730df243bb4d&width=768&dpr=4&quality=100&sign=89a6c6b8&sv=2)

*   La **plataforma** muestra el **canvas** y envía **eventos** a la aplicación en caso de necesitarlo (eventos de ciclo de vida, eventos de cambios de configuración, ...etc. ).

# Características destacables de Flutter | 1. Introducción a Flutter

Flutter transformará el proceso de **desarrollo** y nos permitirá obtener, a partir del **mismo** **código base**, **aplicaciones** **multiplataforma,** **nativas**, **rápidas** y con interfaces de usuario muy **chulas.**

*   Flutter es de **código** **abierto**: puedes ver su evolución, crear tus propios parches, paquetes y contribuir a su desarrollo.
    
*   Flutter dispone de una **excelente** **documentación**.
    
*   Flutter **no** **es** un **lenguaje** de **programación**, es un **framework** de Dart.
    
*   Flutter está **escrito** **C**, **C++** y **Dart.**
    
*   Las **aplicaciones** **Flutter** se programan con **Dart**. Comparte muchas características de otros lenguages modernos como Kotlin y Swift. Además, Dart **traspila** a **Javascript**.
    
*   La **sintaxis** **declarativa** de Flutter para **crear** **interfaces** **de usuario** (utilizando Dart) le permite evitar el código repetitivo (boilerplate code) y ser extremadamente **productivos**.
    

**Paradigmas** que usa Flutter: [https://esflutter.dev/docs/resources/faq#qué-paradigma-de-programación-usa-el-framework-de-flutter](https://esflutter.dev/docs/resources/faq#qu%C3%A9-paradigma-de-programaci%C3%B3n-usa-el-framework-de-flutter)

*   Durante el **desarrollo** de la aplicación, se permite su **recarga** en **caliente** (**hot** **reload**, compilación JIT): las actualizaciones en el código se aplican sin tener que volver a cargar el estado o recompilar la aplicación _\--> los cambios realizados se observan de forma instantánea en la aplicación en ejecución (sin perder el estado actual)._
    
*   La **interfaz** de **usuario** de Flutter es **completamente** **personalizable** (UI customizable). En Flutter prácticamente todo es un **widget**, incluso la propia aplicación es un widget. (widgets para Google Material Design o iOS Cupertino listos para usar (out of the box). Podremos **crear** nuestros propios **widgets**.
    
*   Flutter proporciona excelentes **animaciones** y transiciones.
    
*   Para aquellas **funciones** **específicas** de la plataforma que aún no tienen un paquete de soporte de Flutter: Flutter admite la **interoperabilidad** de **C y C++**, así como canales de plataforma (**platform** **channels**) para interactuar con Kotlin/Java en **Android** y Swift/Objective-C en **iOS**.

Arquitectura de Flutter

Flutter está organizado en **capas** de bibliotecas (libraries) **independientes**, lo cual las hace extensibles (y también reemplazables) donde cada capa depende de la capa inferior.

![](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2F3645852152-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FTJQsPEhsPLjjuhB6XVoa%252Fuploads%252FWuvZsbnIg2BMvYg4J2va%252Fimage.png%3Falt%3Dmedia%26token%3D59845a20-f486-40f7-b0d2-8b725846714c&width=768&dpr=4&quality=100&sign=337eb773&sv=2)

Arquitectura de Flutter

*   **Framework**: programado únicamente en Dart, contiene las bibliotecas con todas las clases necesarias para widgets, rendering, animaciones, gestos, etc. Es la capa con la que el **desarrollador** **Flutter** **interactuará** habitualmente.
    
*   **Engine**: Fundamentalmente programado en C/C++. Proporciona todas las primitivas de bajo nivel para que las aplicaciones Flutter funcionen: **rendering** (**Skia**), **Dart** **runtime**, **channels**, etc.
    
*   **Embedder**: Todas las aplicaciones Flutter se **empaquetan** igual que otra **aplicación** **nativa**. Hay un embedder específico para cada plataforma. Flutter incluye embedders para las plataformas más comunes. Cada embedder se programa con el lenguaje que es adecuado para la plataforma, por ejemplo: Java and C++ para Android, Objective-C/Objective-C++ para iOS y macOS, any C++ for Windows and Linux.

# Instalación rápida | 1. Introducción a Flutter

La forma más **simple** de **descargar, configurar** y **probar** el SDK de Flutter en nuestro sistema es realizarlo a través de **Visual Studio Code**. Luego, **completaremos** la **configuración** para el desarrollo de aplicaciones **Android** con Flutter.

Para ello, realizaremos las siguientes **2** **acciones** contenidas en las secciones desplegables:

1\. Descarga y configuración inicial de Flutter[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/2.-instalacion-del-flutter-sdk/instalacion-rapida#id-1.-descarga-y-configuracion-inicial-de-flutter)

Se ofrece solo un **resumen**, seguiremos las **instrucciones** de la **documentación** **oficial** para nuestro sistema operativo: [https://docs.flutter.dev/get-started/quick](https://docs.flutter.dev/get-started/quick)

**Resumen** de los **pasos**:

1.  **Software necesario:** git y Visual Studio Code, bibliotecas, etc. (si no los tenemos en el sistema, instalarlos)
    
2.  **Instalación y configuración de Flutter:** Se realiza fácilmente a través Visual Studio Code:
    
    1.  Añadimos primero la extensión de Flutter.
        
    2.  Luego, creamos un nuevo proyecto Flutter, donde se detecta que no tenemos el Flutter SDK y comenzará a descargarlo en la carpeta contenedora que hayamos elegido (**se creará una nueva carpeta dentro de la carpeta contenedora**)
        
    3.  Por último, le indicamos que lo añada al PATH del sistema (para tenerlo accesible desde el terminal en cualquier directorio)
        
    
3.  **Ejecución del proyecto Flutter creado:** Probamos la aplicación, que se ejecutará en Chrome (se abrirá automáticamente).
    

2\. Configuración para el desarrollo con Flutter para dispositivos Android.[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/2.-instalacion-del-flutter-sdk/instalacion-rapida#id-2.-configuracion-para-el-desarrollo-con-flutter-para-dispositivos-android)

Para ejecutar, compilar y desplegar las aplicaciones Flutter en dispositivos Android deberemos realizar una serie de configuraciones.

**Deberemos** consultar la **documentación** **oficial** para más detalles: [https://docs.flutter.dev/platform-integration/android/setup](https://docs.flutter.dev/platform-integration/android/setup)

**Resumen** de los **pasos**:

1.  En **Android Studio** (ya instalado previamente)**:**
    
    1.  Realizar las siguientes comprobaciones/acciones en Tools -> SDK Manager (reiniciarlo si se han hecho cambios):
        
        1.  **SDK Platforms:** **API Level** **36** seleccionada.
            
        2.  **SDK Tools:** Han de estar seleccionados:
            
            *   Android SDK Build-Tools
                
            *   Android SDK Command-line Tools
                
            *   Android Emulator
                
            *   Android SDK Platform-Tools
                
            
        
    2.  Instalaremos el **Plugin Flutter** (que instala también el plugin Dart) para desarrollar aplicaciones Android con Flutter en Android Studio.
        
    
2.  **Aceptar las licencias Android:** Antes de poder compilar una aplicación Flutter para Android **deberemos** leer y aceptar las licencias de la plataforma Android SDK.
    
    Para ello, ejecutaremos desde un terminal el comando:
    
    `$ flutter doctor --android-licenses`
    
    Una vez aceptadas, nos informará con el mensaje:
    
    `All SDK package licenses accepted`
    

1.  Deberemos tener un AVD o un Dispositivo físico Android disponibles.
    
2.  **Validar la configuración de Flutter:** Ejecutaremos los siguientes comandos, prestando atención a los possibles errores:
    
    1.  `$ flutter doctor`
        
    2.  `$ flutter emulators && flutter devices`
        
    

Con todos los pasos anteriores ya podremos ejecutar, depurar y desplegar aplicaciones Flutter en dispositivos Android. El **desarrollo** lo podremos realizar desde Visual Studio Code o con Android Studio/IntelliJ (con el plugin de Flutter instalado).

#### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/2.-instalacion-del-flutter-sdk/instalacion-manual/ejemplo-flutter-in-ubuntu-step-by-step.#id-1-downloading-vs-code)

*   You can **download** VS Code:
    
    *   From [here](https://code.visualstudio.com/)
        . It doesn’t have much download size and is easy to set up.
        
    *   You can also use the terminal to install the VS Code on your system, by running the following code : `$ sudo apt install code`
        
    

#### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/2.-instalacion-del-flutter-sdk/instalacion-manual/ejemplo-flutter-in-ubuntu-step-by-step.#id-2-downloading-flutter-sdk)

After installing VS Code, we need to **install Flutter SDK** so that it will enable VS Code to build flutter applications.

In order to install the flutter SDK, you have to follow instructions from the [Flutter Website](https://flutter.dev/docs/get-started/install/linux)

Once you’re done installing the VS Code and Flutter SDK, let’s try opening VS Code. If there hadn’t been any installation errors, VS Code will open without any issue.

After loading up, we need to **enable** **Git** which is a version control system (VCS) which will be really helpful in the later stages.

For this, select VCS Option from the left side of the VS Code. Please refer to the below image if you’re new to VS Code.

![VCS option](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2Fmedia.geeksforgeeks.org%2Fwp-content%2Fuploads%2F20200823113253%2Fs2-660x436.png&width=300&dpr=4&quality=100&sign=ba4f3796&sv=2)

After selecting that, you will find an option to install Git. Once you click on that, you will be taken to the website from which you can download Git for your system. You can also install Git by running the following code on your terminal.

Copy

    $ sudo apt update
    
    $ sudo apt install git

After Installing Git, come back to the VS Code and select the **Extensions** (the icon present at two down the position with that of VCS).

![installing git](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2Fmedia.geeksforgeeks.org%2Fwp-content%2Fuploads%2F20200823114235%2Fs34-660x436.png&width=300&dpr=4&quality=100&sign=5fcaf432&sv=2)

From that, you have to **search** **for** **flutter** **and** **install** that extension. Also, **make sure that Dart is also installed i**n the same method.

![flutter plugin](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2Fmedia.geeksforgeeks.org%2Fwp-content%2Fuploads%2F20200823114314%2Fs341-660x436.png&width=300&dpr=4&quality=100&sign=aca6daf7&sv=2)

Now you’re nearly ready to go. There is one more final installation that you need to perform before you’re all set to go. You will have to install Android Studio for all the essential packages like Manager or if you need to add any virtual devices via AVD Manager.

#### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/2.-instalacion-del-flutter-sdk/instalacion-manual/ejemplo-flutter-in-ubuntu-step-by-step.#id-4-installing-android-studio)

**4) Installing Android Studio**

Though we might be able to download all the packages separately, it may lead on to some errors. To avoid that, let’s install Android Studio so that all the essential packages will get added correctly. You can install Android Studio by running the following command on the terminal.

*   First **install** **JDK** :
    

Copy

    $ sudo apt install openjdk-17-jdk

*   Now, you can **install** **Android** **Studio from** [https://developer.android.com/studio/install](https://developer.android.com/studio/install)
     or:
    

Copy

    $ sudo snap install android-studio --classic

After installing Android Studio, open it from the Applications directory and run it. You will be asked whether to import any settings and select ‘Do not import’ for that. Move ahead with the installation flow and finally, all the installation will be over.

After the installation is over, you will reach the **starting** **page**. From there select the ‘**Configure**’ option present on the right side bottom.

![configure android studio](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2Fmedia.geeksforgeeks.org%2Fwp-content%2Fuploads%2F20200823120640%2Fs56-660x436.png&width=300&dpr=4&quality=100&sign=b86908ed&sv=2)

Now, you have to **select** ‘**SDK** **Manager**’ from the list and you will be redirected to a page where you can install the required Android SDKs for various Android OS Versions. **Please make sure that you’re installing the SDK of the same version as that of your phone or the virtual device which you will be setting up later**. Select them and **click** on ‘**Apply**’. After that, you will be prompted to install them and follow the procedure.

![select android version](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2Fmedia.geeksforgeeks.org%2Fwp-content%2Fuploads%2F20200823121015%2Fs561-660x436.png&width=300&dpr=4&quality=100&sign=7fc2a9ed&sv=2)

Once everything is complete, **close** **everything** including **Android** **Studio** as well as **VS** **Code**. You can **restart VS Code now.** Within VS Code, you can create a new workspace folder (where your project files will be stored). After that, click on View from the top Tool Bar and select the Command Panel.

![](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2Fmedia.geeksforgeeks.org%2Fwp-content%2Fuploads%2F20200823122614%2Fs563-660x436.png&width=768&dpr=4&quality=100&sign=64845451&sv=2)

In that type in Flutter New Project and select it. If everything has been set up correctly, you will be asked to type in your project name. Sometimes, you will be asked to locate the Flutter SDK. In that case, provide the location where you installed the Flutter SDK.

If there were no errors, a project of the name that you provided will load up in a moment. The same can also be achieved by running the following command on the terminal present on VS Code (select Terminal from the toolbar on the top and run the following command.

`flutter create <projectname>`

where <projectname> can be changed with the name of your preference. Once the project loads completely, all the files can be accessed from the Files section of VS Code present on the left side tool bar (2 positions above that of the VCS).

You may connect your physical device (android) for installing the application or you may install a new Android Virtual Device from the AVD Manager, just like how we installed the SDKs from SDK Manager. The Run option can be found just below VCS and it can be used to run and debug your application.

Now, you’re all set to develop flutter application using VS Code.

Happy Coding!

# Hot Reload - Hot Restart - Full Restart | 1. Introducción a Flutter

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/3.-primera-aplicacion-en-flutter/hot-reload-hot-restart-full-restart#hot-reload)

**Hot reload ⚡️**

Carga los cambios del código en la VM o en el navegador y reconstruye el árbol de widgets, **preservando** el **estado** **de la aplicación**; **pero NO ejecuta de nuevo** `**main()**` **o** `**initState()**`**.** (`⌘\` o `CTRL\` en Intellij and Android Studio, `⌃F5` en VSCode)

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/3.-primera-aplicacion-en-flutter/hot-reload-hot-restart-full-restart#hot-restart)

**Hot restart** 🔄

Carga los cambios del código en la VM o en el navegador, y **reinicia** la **aplicación**, por tanto, **se** **pierde** **el** **estado** **de la aplicación**. En el navegador, permite reiniciar la aplicación sin tener que recargar la página. (`⇧⌘\` o `⇧CTRL\` en IntelliJ and Android Studio, `⇧⌘F5` en VSCode)

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/3.-primera-aplicacion-en-flutter/hot-reload-hot-restart-full-restart#full-restart)

**Full restart** ⏹️

*   **Reinicia por completo** la **aplicación** iOS, Android, o web. Esta operación es lenta pues **recompila** **de nuevo** el código Java / Kotlin / Objective-C / Swift. En la web, también reinicia el Dart Development Compiler. (No hay un atajo de teclado, se debe parar y volver ejecutar la aplicación)
    
*   Ocurre **siempre** que **paramos** la aplicación y la **ejecutamos** de nuevo.
    

Más información en: [https://docs.flutter.dev/tools/hot-reload](https://docs.flutter.dev/tools/hot-reload)

# Widgets | 1. Introducción a Flutter

Catálogo de widgets: [https://docs.flutter.dev/development/ui/widgets](https://docs.flutter.dev/development/ui/widgets)

Para practicar: [https://docs.flutter.dev/development/ui/interactive](https://docs.flutter.dev/development/ui/interactive)

En Flutter los elementos de la UI son objetos del tipo **Widget** (casi todo en una aplicación Flutter son widgets)

El **widget** describe cómo debe mostrarse el elemento UI en pantalla teniendo en cuenta la actual **configuración** y el **estado**.

Flutter proporciona un **catálogo** muy completo de **widgets** **standard** de todo tipo: botones, listas, pilas, ....

Los widgets se pueden **personalizar** utilizando las propiedades y pueden contener uno o más hijos. Así, por ejemplo, si queremos un widget que muestre un texto personalizado, crearemos un nuevo widget que se **componga** de Text.

*   Flutter prefiere la **composición** sobre la herencia. Toda app Flutter se compone de ventanas que a su vez se componen de widgets: **app -> screens -> widgets**
    

Los widgets están diseñados para ser **inmutables** (unchangeable) lo que ayuda a mantener la UI muy ligera (para actualizar la UI se deberá crear una nueva instancia con los cambios: Flutter se encarga de reutilizar y recyclar de forma eficiente los widgets)

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/widgets#tipos-de-widgets)

Tipos de Widgets

Nuestras aplicaciones se comprondrán de widgets que son **subclase** de:

*   `**StatelessWidget**`: Widgets que solo dependen para su configuración de la información **estática** que contienen (por ejemplo, una imagen estática en un image view, o un texto de título).
    
*   `**StatefulWidget**`: Widgets que necesitan mantener información de **estado** dinámica. Lo consiguen utilizando un objeto del tipo `State`.
    

Ambos tipos de **widgets**, `stateless` y `stateful` **se redibujan cada vez que el framework de Flutter se lo pide, por tanto,** **todo widget deberá implementar el método** `**build()**` (que describe el widget a partir de otros widgets, configurando así el **árbol de widgets**)

La **diferencia** entre ambos tipos de widgets es que los widgets `**stateful**` delegan su configuración a un objeto `**State**` y se pueden **actualizar** a **petición** en tiempo de ejecución.

# Stateless widget | 1. Introducción a Flutter

Los widgets **stateless** extienden de la clase `**StatelessWidget**` y son **inmutables** y, por ello, una vez creados, ya no van a cambiar sus propiedades durante la ejecución de la aplicación.

*   Todos sus atributos deberían declararse _final_ (aunque Dart no obliga)
    
*   Son widgets que se optimizan internamente por el runtime de Flutter
    
*   Deben de utilizarse siempre que el estado del widget no vaya a cambiar, es decir, depende únicamente de sus datos internos.
    

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/widgets/stateless-widget#ejemplo-1)

**Ejemplo 1**

Creamos una aplicación MaterialApp compuesta por un **stateless** **widget:** **HomePage**, donde **count** es un atributo entero declarado como _final_ que no cambiará en tiempo de ejecución.

Copy

    import 'package:flutter/material.dart';
    
    void main() => runApp(MyApp());
    
    class MyApp extends StatelessWidget {
      @override
      Widget build(BuildContext context) {
        return const MaterialApp(
          home: Homepage(count: 10),
        );
      }
    }
    
    class Homepage extends StatelessWidget {
      final int count;
    
      const Homepage({required this.count});
    
      @override
      Widget build(BuildContext context) {
        return Text('Count = $count');
      }
    }

      @override

            appBar: AppBar(
              title: const Text('Welcome to Flutter'),

Vista del proyecto Flutter desde Android Studio

*   `**.dart_tool/**` - Directorio autogenerado tras ejecutarse _pub get_. Contiene el fichero _package\_config.json_ que especifica la versión de cada paquete y su localización.
    
*   `**.idea/**` - Directorio que contiene la configuración de Android Studio para el proyecto.
    
*   `**android/**` - Directorio que contiene el código autogenerado para el proyecto Android cuando estamos desarrollando la aplicación flutter para Android. Cuando el proyecto Flutter se compila a código nativo, se inyectará dentro de este proyecto Android para generar la aplicación Android nativa.
    
*   `ios/` - Directorio que contiene el código autogenerado para el proyecto iOS mientras desarrollamos la aplicación Flutter para iOS. Cuando el proyecto Flutter se compila a código nativo, se inyectará dentro de este proyecto iOS para generar la aplicación iOS nativa.
    

> Para compilar la aplicación Flutter para iOS se precisará de macOS. Por ello, este directorio sólo estará disponible cuando el proyecto se cree en un Mac.

*   `web/` - Este directorio autogenerado contiene el código del proyecto Web cuando creamos la aplicación Flutter para web. Cuando el proyecto Flutter se compila a código nativo, se inyectará dentro de este proyecto web para generar la aplicación web nativa.
    
*   `**build/**` \- Directorio autogenerado que contiene el código compilado de la aplicación Flutter. Se actualiza de forma incremental por motivos de rendimiento (compilar cada vez por completo la aplicación tomaría mucho tiempo)
    
*   `**lib/**` - (library) Este es el directorio principal cuando estamos creando nuestra aplicación Flutter. Contendrá todos los ficheros Dart que utilizan el framework de flutter. Por defecto, contiene el fichero `**main.dart**` que es el punto de entrada de la aplicación Flutter.
    
*   `test/` - Directorio con el código Dart para los tests automáticos de nuestra aplicación Flutter. El fichero _widget\_test.dart_ contiene código de test de ejemplo.
    
*   `.gitignore` - Fichero que contiene la lista de ficheros y directorios que serán ignorados por Git.
    
*   `.metadata` - Fichero autogenerado por las herramientas de flutter que contiene metadatos o propiedades de nuestra aplicación flutter.
    
*   `helloflutter.iml` - Fichero con el mismo nombre del proyecto. Contiene configuraciones adicionales del proyecto que utiliza Android studio.
    
*   `**pubspec.yaml**` - Fichero de configuración del proyecto Flutter que utiliza Pub, la herramienta de gestión de paquetes de Flutter.
    
*   `pubspec.lock` - Fichero autogenerado por Pub (Flutter package manager) a partir del fichero pubspec.yaml que indica las versiones de los paquetes utilizadas en el proyecto.
    
*   `README.md` - Fichero autogenerado al crear el proyecto que contiene una descripción inicial.

*   **Linea (5):** La **clase** **MyApp** es un widget stateless y se usa para crear la UI. Esta clase ha de sobrescribir el método _**build**_ el cual será llamado por el framework de Flutter para construir la UI de la aplicación.

*   **Linea (7):** El método sobrescrito `**build**` devuelve un widget del tipo _**MaterialApp**_ que crea la UI principal de la aplicación. En este caso, tiene definida la propiedad `**home**` que specifica la **UI** de la aplicación con el widget: `**Scaffold:**`

Organización de los directorios: estructura propuesta


-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

*   No hay una forma estandarizada para estructurar nuestra aplicación.
    
*   Pero, a medida que un proyecto Flutter crezca deberemos de tener una estructura clara y organizada de todos sus ficheros de forma que se facilite el trabajo en equipo así como el mantenimiento de la aplicación (no debemos perder tiempo buscando un widget o viendo dónde situar uno nuevo).

      |_ videos
      |_ fonts

      |_ cloud_functions  // código para obtener datos de servidores (ej: abrir URL)

                          // código UI: para las pantallas de nuestra aplicación 
                          // se puede organizar en subdirectorios: login, home, others
    
      |_ providers        // código que transforma los datos recibidos desde el exterior (network)

      |_ models           // código para los datos que provienen de servidores, del usuario, etc.

Import relativo vs import package


----------------------------------------------------------------------------------------------------------------------------------------------------------------------------

La convención y **mejores** **prácticas** en Flutter prefiere usar **imports** **relativos** al package **para** **código** **dentro** de tu **propio** proyecto aunque tenemos varias **alternativas**:

1.  **Imports dentro de tu proyecto (lib/):**
    

Copy

    // ✅ EL MEJOR: Usar imports relativos para archivos en lib/
    import 'screens/home_screen.dart';
    import 'widgets/custom_button.dart';
    import 'utils/constants.dart';

1.  **Imports de paquetes externos:**
    

Copy

    // ✅ CORRECTO: Usar package: para dependencias externas
    import 'package:flutter/material.dart';
    import 'package:provider/provider.dart';

1.  **Imports de tu propio paquete:**
    

Copy

    // ✅ TAMBIÉN VÁLIDO: Usar package: para tu propio código
    import 'package:mi_app/screens/home_screen.dart';
    import 'package:mi_app/widgets/custom_button.dart';

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/5.-recomendaciones-best-practices#por-que-se-prefieren-los-imports-relativos-para-codigo-de-tu-proyecto)

¿Por qué se prefieren los imports relativos para código de tu proyecto?


------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Razones:

*   Más **cortos** y **limpios**
    
*   Más **fáciles** de **mover** archivos (**refactoring**)
    
*   **Menor** probabilidad de **errores** al renombrar el package
    
*   Es la **convención** **más** **común** en la comunidad Flutter
    

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/5.-recomendaciones-best-practices#cuando-usar-package)

¿Cuándo usar `package?`


----------------------------------------------------------------------------------------------------------------------------------------------------

1.  Siempre para importar **dependencias** **externas**
    
2.  Cuando importas desde **test/** código de lib/
    

Ejemplo:[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/5.-recomendaciones-best-practices#ejemplo)

Copy

    mi_proyecto/
    ├── lib/
    │   ├── src/
    │   │   └── calculator.dart
    │   └── main.dart
    └── test/
        └── calculator_test.dart
    
    // En calculator_test.dart DEBES usar package:
    import 'package:mi_proyecto/src/calculator.dart';  // ✅ CORRECTO
    // NO puedes usar:
    import '../lib/src/calculator.dart';  // ❌ INCORRECTO

Esto es porque los archivos en `test/` están fuera del directorio `lib/` y necesitan referenciar el código como si fuera un paquete.

1.  Cuando tienes **múltiples** **packages** en el mismo proyecto.
    

Ejemplo[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/5.-recomendaciones-best-practices#ejemplo-1)

    │   │   └── lib/
    │   │       └── feature_a.dart
    │   └── feature_b/
    │       └── lib/
    │           └── feature_b.dart
    └── main_app/
        └── lib/
            └── main.dart
    
    // En main.dart cuando quieres usar feature_a:
    import 'package:feature_a/feature_a.dart';  // ✅ CORRECTO

Esto es común en **proyectos** **grandes** donde divides funcionalidades en packages separados dentro del mismo proyecto.

Explicación: [https://medium.flutterdevs.com/app-lifecycle-in-flutter-c248d894b830](https://medium.flutterdevs.com/app-lifecycle-in-flutter-c248d894b830)

**createState()**: Se llama cuando creamos un nuevo StatefulWidget. Se encarga de crear el estado mutable para este widget instanciando el objeto State asociado a este widget.

**initState()**: Se llama después del constructor y sólo una vez para cada objecto State. Aquí podremos inicializar datos, listeners, etc.

**didChangeDependencies()**: Se llama después de initState() y también cuando se actualiza algún objeto del cual dependa, por ejemplo un InheritedWidget u otras dependencias (cambio de tema, orientación del dispositivo, etc.)

> build() se ejecutará de forma inmediata, así que raramente lo implementaremos. A no ser que tengamos un proceso costoso, como obtener datos de un servidor en red, y que no se ha de hacer en cada build(), sino sólo cuando cambia el InheritedWidget.

**build()**: Devuelve el Widget que se mostrará en pantalla. Se llama de forma frecuente (después de iniState(), tras setState(),....) cuando se necesita renderizar los UI widgets en pantalla. Este método solo debe encargarse de construir el widget, nada más. El framework se encargará de optimizar su inserción en el árbol de widgets.

**didUpdateWidget**(): Se llama siempre que se cambia el Widget asociado a este objeto State por un nuevo Widget del mismo _runtimeType_ y _key_ que el anterior. El framework asocia State con el nuevo Widget. Después de este método, se llama siempre a _build()._ _\--> Deberemos sobrescribir didUpdateWidget para actualizar correctamente el estado con los datos del nuevo widget que se ha creado (o seguiremos mostrando el estado con los datos del anterior widget)_

[https://api.flutter.dev/flutter/widgets/State/didUpdateWidget.html](https://api.flutter.dev/flutter/widgets/State/didUpdateWidget.html)

"If the parent widget rebuilds and request that this location in the tree update to display a new widget with the same [runtimeType](https://api.flutter.dev/flutter/widgets/State/dart-core/Object/runtimeType.html)
 and [Widget.key](https://api.flutter.dev/flutter/widgets/State/widgets/Widget/key.html)
, the framework will update the [widget](https://api.flutter.dev/flutter/widgets/State/widgets/State/widget.html)
 property of this [State](https://api.flutter.dev/flutter/widgets/State/widgets/State-class.html)
 object to refer to the new widget and then call this method with the previous widget as an argument."

**setState()**: Es la forma de notificar al framework de Flutter que los datos, el estado, han cambiado y que el widget debe reconstruirse. Este método toma como argumento una función callback que se encarga de actualizar los datos (ha de ser un proceso rápido).

**deactivate()**: Se llama cuando momentáneamente se elimina este objeto State del árbol de widgets (los objetos State pueden moverse de lugar en el árbol), però podría ser reinsertado antes de que acabe el _frame_ actual.

**dispose()****:** Se llama cuando este objeto State se va a eliminar de forma permanente del árbol de widgets. En este método es donde deberíamos cancelar animaciones, streams, cerrar conexiones, etc.

# Stateful widget | 1. Introducción a Flutter

Un widget **stateful** extiende de la clase `**StatefulWidget**`. Contiene datos que pueden **cambiar** en tiempo de ejecución y que afectarán a la **visualización** de la UI, que deberá ser actualizada debidamente.

Los **StatefulWidgets** resultan útiles cuando la parte de la UI que estamos describiendo puede cambiar. Por ejemplo: el usuario pulsa un botón y el texto de un widget Text debe actualizarse.

El **estado** del widget viene dado por sus propiedades (datos).

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/widgets/stateful-widget#implementacion-de-un-statefulwidget)

Implementación de un StatefulWidget


------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Para **implementar** un S_tateful widget_, necesitamos definir dos clases:

*   Una clase que extiende de la clase `**StatefulWidget**` que creará una instancia de la clase `State.`
    
*   Una clase que extiende de `**State**` que contiene:
    
    *   El **estado** _**mutable**_ para este widget
        
    *   El método `**build()**`
        
    

Desde el objeto `**State**` podremos acceder a BuildContext directamente con la **propiedad** `_**context**_`, así como a su widget asociado con la **propiedad** `_**widget**_`.

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/widgets/stateful-widget#como-actualizar-el-estado-del-widget-y-hacerlo-visible-en-pantalla)

¿Cómo actualizar el estado del widget y hacerlo visible en pantalla?

Cada vez que **actualicemos** las propiedades del widget, es decir, su **estado**, deberemos llamar al método `**setState(..)**` que indica al framework de Flutter que debe **redibujar** de nuevo el widget.

`setState(..)` provocará que el framework de Flutter llame de nuevo al método `**build()**` de este objeto `State` el cual devolverá un **nuevo** **widget** con los datos **actualizados** (**nuevo** **estado**) y que ya podremos ver en pantalla.

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/widgets/stateful-widget#ejemplo-1-widget-con-estado-para-mostrar-un-contador)

**Ejemplo 1: Widget con estado para mostrar un contador**


---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Vamos a crear una aplicación `MaterialApp` compuesta por **widget** **con** **estado** que contiene un botón, que ocupará toda la pantalla, y que muestra un texto con el valor actual de un contador.

**Funcionamiento**:

*   Al pulsar sobre el botón se incrementará el valor del contador.
    
*   Pero, si queremos ver en pantalla el contador actualizado, se debe **ejecutar** el método `**setState(...)**` que provoca que el framework de Flutter renderize de nuevo el botón (se llamará de nuevo a build()) mostrando así en pantalla el nuevo valor del contador.
    

![](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2F3645852152-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FTJQsPEhsPLjjuhB6XVoa%252Fuploads%252FlDKIpCXYqamMnAO5iRlN%252Fimage.png%3Falt%3Dmedia%26token%3D60f0646c-8f11-4d18-85dd-74eaba5481a6&width=768&dpr=4&quality=100&sign=2ee5a62&sv=2)

El **código** sería:

Copy

    import 'package:flutter/material.dart';
    
    void main() => runApp(App());
    
    class App extends StatelessWidget {
      @override
      Widget build(BuildContext context) {
        return MaterialApp(
          home: Homepage(),
        );
      }
    }
    class Homepage extends StatefulWidget {
      @override
      State<Homepage> createState() => _HomepageState();
    }
    
    class _HomepageState extends State<Homepage> {
      int _count = 0;
    
      @override
      Widget build(BuildContext context) {
        return ElevatedButton(
          child: Text('Pressed: $_count'),
          onPressed: _increaseCounter,   // o directamente: () => setState(() => count++),
        );
      }
      void _increaseCounter() => setState(() => _count++);
    }

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/widgets/stateful-widget#explicacion-del-codigo)

**Explicación del código**:

*   `**State<Homepage>:**` La clase _state_ extiende de `State<T>` con `T` siendo del tipo del correspondiente `StatefulWidget`. La clase `state` creada es exclusiva para `T` y no se puede reutilizar en otro `StatefulWidget`
    

El propio widget `Stateful` es **inmutable**, es su objeto `State` el que persiste durante el tiempo de vida del widget.

*   Cuando queremos cambiar el estado del widget, debemos llamar a _**setState**__():_ indicamos así al framework que redibuje (re-render) el widget, se llamará al `**build**` de este widget (y el de todos los widgets del árbol que se puedan ver implicados).
    

> **setState(...)**: toma como argumento una función del tipo: `VoidCallback`
> 
> Un widget stateless puede contener widgets stateful y viceversa.

*   `**build():**` devuelve un `ElevatedButton` que tiene asignado en `onPressed` el método _\_increaseCounter_ que se llamará al pulsarlo. _\_increaseCounter_ se encargará de llamar a setState() desde donde se incrementa el contador.
    

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/widgets/stateful-widget#por-que-se-necesita-un-objeto-state-separado-para-un-statefulwidget)

¿Por qué se necesita un objeto _State_ separado para un _StatefulWidget_?

La razón es porque en Flutter estos objetos tienen ciclos de vida diferentes:

*   Los `Widgets` son objetos **temporales**, utilizados para construir una visualización (UI) de la aplicación en su estado actual.
    
*   Los objetos `State`, por otra parte, son **persistentes** entre diferentes llamadas a `build()`, de esta forma, los widgets pueden recordar la información (**mantienen su estado entre llamadas a build()**).
    

> En aplicaciones más complejas, cada widget de la jerarquía tiene unas responsabilidades concretas; por ejemplo, un widget podría representar la interfaz de usuario para recoger una información específica (por ejemplo, fecha y lugar) mientras que otro widget se encargaria de usar esa información para cambiar la visualización de la UI.

En Flutter, las **notificaciones** **de** **cambio** **fluyen** **hacia** "**arriba**" de la jerarquía de widgets, mientras que la **visualización** **del** **estado** **actual** **fluye** hacia "**abajo**", hacia los widgets stateless que se encargan de visualizar la UI (en el estado actual que tenga el widget). El objeto común que redirige este flujo es el `**State**`.

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/widgets/stateful-widget#ejemplo-2-flujos-de-notificaciones-de-actualizacion)

**Ejemplo 2: Flujos de notificaciones de actualización**

Veamos un ejemplo más complejo del **contador** donde se vean cómo funcionan los flujos de notificación y de visualización (re-rendering).

![](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/~gitbook/image?url=https%3A%2F%2F3645852152-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FTJQsPEhsPLjjuhB6XVoa%252Fuploads%252F38D94FPDQZTgY7HC8sko%252Fimage.png%3Falt%3Dmedia%26token%3D3dedbab6-b98b-47cd-b2e6-269c41a8be15&width=300&dpr=4&quality=100&sign=3f744072&sv=2)

La **aplicación** **se** **compone** de los siguientes widgets:

*   Dos widgets _stateless_, cada uno cumple su función, separando responsabilidades:
    
    *   `CounterDisplay` : Muestra un texto con el valor del contador:
        
    *   `CounterIncrementor` : Contiene un botón, que al pulsarlo, llamará a una función (callback) que incrementará el valor del contador.
        
    
*   El widget _stateful_ `Counter` se compone de los dos widgets stateless anteriores y contiene el estado: el valor del contador (que se incrementará al pulsar el botón).
    

El resultado es similar al Ejemplo 1 pero al **separar** **responsabilidades** se permite que la **complejidad** se **encapsule** en los widgets individuales a la vez que se mantiene la **simplicidad** en el widget padre `Counter`.

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/widgets/stateful-widget#veamos-el-codigo)

Copy

    import 'package:flutter/material.dart';
    
    void main() {
      runApp(
        const MaterialApp(
          home: Scaffold(
            body: Center(
              child: Counter(),
            ),
          ),
        ),
      );
    }
    
    class CounterDisplay extends StatelessWidget {
      const CounterDisplay({required this.count, Key? key}) : super(key: key);
    
      final int count;
    
      @override
      Widget build(BuildContext context) {
        return Text('Count: $count');
      }
    }
    
    class CounterIncrementor extends StatelessWidget {
      const CounterIncrementor({required this.onPressed, Key? key}) : super(key: key);
    
      final VoidCallback onPressed;  // VoidCallback type: no args and no returned data
    
      @override
      Widget build(BuildContext context) {
        return ElevatedButton(
          onPressed: onPressed,
          child: const Text('Increase Counter'),
        );
      }
    }
    
    class Counter extends StatefulWidget {
      const Counter({Key? key}) : super(key: key);
    
      @override
      State<Counter>  createState() => _CounterState();
    }
    
    class _CounterState extends State<Counter> {
      int _counter = 0;
    
      // setState(..): makes that framework calls build()
      void _increment() => setState(() => ++_counter);  
    
      @override
      Widget build(BuildContext context) {
        return Column(
          mainAxisAlignment: MainAxisAlignment.center,
          children: <Widget>[\
            CounterDisplay(count: _counter),\
            const SizedBox(height: 16, ),    // to make same room between widgets\
            CounterIncrementor(onPressed: _increment),   // set callback\
          ],
        );
      }
    }

# Ciclo de vida de la aplicación | 1. Introducción a Flutter

En una aplicación Flutter, tendremos acceso a una selección de los eventos del ciclo de vida de la aplicación (en una app nativa tendríamos más eventos).

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/ciclo-de-vida/ciclo-de-vida-de-la-aplicacion#estados)

Estados


-------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Los diferentes **estados** en los que puede estar la aplicación (ciclo de vida) vienen determinados por el `enum` [`AppLifecycleState`](https://api.flutter.dev/flutter/dart-ui/AppLifecycleState.html)
 y son los siguientes:

*   **detached**: La aplicación **no tiene interfaz visible**, está en proceso de creación/destrucción.
    
*   **resumed**: La aplicación está **visible** e **interactuando** con el usuario. (_Resumed_ de Android)
    
*   **inactive**: La aplicación es **visible** **pero** **no** **recibe** **interacción** del usuario: split-screen, un diálogo del sistema aparece, una llamada de teléfono,.. (Sería similar a un estado _Paused_ de Android)
    
*   **hidden:** La aplicación **no** es **visible,** está a punto de ser pausada.
    
*   **paused**: La aplicación **no** es **visible**, está en segundo plano, y **no** **interactúa** con el usuario. ( _Stopped_ de Android)
    

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/ciclo-de-vida/ciclo-de-vida-de-la-aplicacion#eventos-del-ciclo-de-vida)

Eventos del ciclo de vida


-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/ciclo-de-vida/ciclo-de-vida-de-la-aplicacion#primera-forma-widgetsbindingobserver)

Primera forma: `**WidgetsBindingObserver**`

Para **capturar** los **eventos** del ciclo de vida de la aplicación:

*   Utilizaremos la **clase** **abstracta** `**WidgetsBindingObserver**` como _**mixin**_ que nos permitirá observar los eventos del ciclo de vida y sobrescribir solo los métodos que nos interesen (el mixin no obliga a sobrescribir todos los métodos, al contrario que implementar una interfaz)
    
*   **Sobrescribiremos** el método `**didChangeAppLifecycleState(...)**` que se llamará cuando haya un cambio de estado en la aplicación.
    

Ejemplo 1: Código[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/ciclo-de-vida/ciclo-de-vida-de-la-aplicacion#ejemplo-1-codigo)

Copy

    import 'package:flutter/material.dart';
    
    void main() {
      runApp(MyApp());
    }
    class MyApp extends StatelessWidget {
    
      @override
      Widget build(BuildContext context) {
        return MaterialApp(
          home: MyHomePage(),
        );
      }
    }
    
      class MyHomePage extends StatefulWidget {
      @override
      _MyHomePageState createState() => _MyHomePageState();
    }
    
    class _MyHomePageState extends State<MyHomePage>
        with WidgetsBindingObserver {
    
      @override
      void initState() {
        WidgetsBinding.instance.addObserver(this);
        super.initState();
      }
    
      @override
      void dispose() {
        WidgetsBinding.instance.removeObserver(this);
        super.dispose();
      }
    
      // WidgetsBindingObserver.
      // Called when the system puts the app in the background 
      // or returns the app to the foreground.
      @override
      void didChangeAppLifecycleState(AppLifecycleState state) {
        print('State: ${state.name}');
      }
    
      @override
      Widget build(BuildContext context) {
        return Scaffold(
          appBar: AppBar(
            title: Text('Flutter - Lifecycle'),
          ),
          body: Center(),
        );
      }
    }

**WidgetsBinding** és la **clase** **central** que actúa com el **vínculo** entre el framework Flutter y el motor Flutter subyacente. Es un **objeto** **singleton**, `WidgetsBinding.instance`y se encarga, entre otros de la supervisión del ciclo de vida, coordinar el renderizado correcto de los widgets, hacer llegar los eventos de entrada como gestos y toques a los widgets adecuados, accesibilidad, etc.

Los **cambios** de **estado** los podemos observar en el **Logcat** cuando, por ejemplo realizamos las siguientes acciones:

1.  Dejamos la aplicación en segundo plano: **inactive -> hidden -> pause**
    
2.  Luego, volvemos a ella: **hidden -> inactive ->resumed**
    
3.  Finalmente, la cerramos: **inactive ->** **hidden -> paused ->detached**

      runApp(MyApp());

        return MaterialApp(
          home: MyHomePage(),
        );
      }
    }

      @override
      _MyHomePageState createState() => _MyHomePageState();

        super.initState();

          onDetach: () => print('Event: onDetach'),
          onShow: () => print('Event: onShow'),

          onStateChange: (state) => print('State: ${state.name}'),

      }
    
      @override
      void dispose() {
        _myAppStateListener.dispose();
        super.dispose();
      }
    
      @override
      Widget build(BuildContext context) {
        return Scaffold(
          appBar: AppBar(
            title: Text('Flutter - Lifecycle - AppLifecycleListener'),
          ),
          body: Center(
              child: Text('To observe states and lifecycle events: \n'
              'Change device orientation, hide application, return back...\n ')
          ),
        );
      }
    }

# Responding to widget lifecycle events, Keys | 1. Introducción a Flutter

[https://docs.flutter.dev/development/ui/widgets-intro](https://docs.flutter.dev/development/ui/widgets-intro)

[https://www.raywenderlich.com/22416843-unlocking-your-flutter-widgets-with-keys](https://www.raywenderlich.com/22416843-unlocking-your-flutter-widgets-with-keys)

After calling [`createState()`](https://api.flutter.dev/flutter/widgets/StatefulWidget-class.html#createState)
 on the `StatefulWidget`, the framework inserts the new state object into the tree and then calls [`initState()`](https://api.flutter.dev/flutter/widgets/State-class.html#initState)
 on the state object. A subclass of [`State`](https://api.flutter.dev/flutter/widgets/State-class.html)
 can override `initState` to do work that needs to happen just once. For example, override `initState` to configure animations or to subscribe to platform services. Implementations of `initState` are required to start by calling `super.initState`.

When a state object is no longer needed, the framework calls [`dispose()`](https://api.flutter.dev/flutter/widgets/State-class.html#dispose)
 on the state object. Override the `dispose` function to do cleanup work. For example, override `dispose` to cancel timers or to unsubscribe from platform services. Implementations of`dispose` typically end by calling `super.dispose`.

For more information, see [`State`](https://api.flutter.dev/flutter/widgets/State-class.html)
.

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/ciclo-de-vida/responding-to-widget-lifecycle-events-keys#keys)

Keys

Use keys to control which widgets the framework matches up with other widgets when a widget rebuilds. By default, the framework matches widgets in the current and previous build according to their [`runtimeType`](https://api.flutter.dev/flutter/widgets/Widget-class.html#runtimeType)
 and the order in which they appear. **With keys, the framework requires that the two widgets have the same** [`**key**`](https://api.flutter.dev/flutter/foundation/Key-class.html)
 **as well as the same** `**runtimeType**`**.**

**Keys are most useful in widgets that build many instances of the same type of widget**. For example, the i a `ShoppingList` widget, which builds just enough `ShoppingListItem` instances to fill its visible region:

*   Without keys, the first entry in the current build would always sync with the first entry in the previous build, even if, semantically, the first entry in the list just scrolled off screen and is no longer visible in the viewport.
    
*   By assigning each entry in the list a “semantic” key, the infinite list can be more efficient because the framework syncs entries with matching semantic keys and therefore similar (or identical) visual appearances. Moreover, **syncing the entries semantically means that state retained in stateful child widgets remains attached to the same semantic entry rather than the entry in the same numerical position in the viewport**.
    

For more information, see the [`Key`](https://api.flutter.dev/flutter/foundation/Key-class.html)
 API.

### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/ciclo-de-vida/responding-to-widget-lifecycle-events-keys#global-keys)

Types of keys

There are two types of keys in Flutter: `GlobalKey`s and `LocalKey`s.

*   Local keys are unique only among siblings.
    
*   Global keys are globally unique across the entire widget hierarchy.
    

#### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/ciclo-de-vida/responding-to-widget-lifecycle-events-keys#the-different-types-of-localkeys-are)

The different types of `LocalKey`s are:

*   _ValueKey_: A key that uses a simple value such as a String.
    
*   _ObjectKey_: A key that uses a more complex data format rather than a primitive data type like a String.
    
*   _UniqueKey_: A key that is unique only to itself. Use this type of key when you don't have any other data that makes a widget unique when using a `ValueKey` or an `ObjectKey`.
    

#### 

[](https://ecm-pmdm-flutter.gitbook.io/1.-introduccion-a-flutter/4.-conceptos-fundamentales-en-flutter/ciclo-de-vida/responding-to-widget-lifecycle-events-keys#global-keys-1)

Global keys

Use global keys to uniquely identify child widgets. , unlike local keys which need only be unique among siblings.

Because they are globally unique, a global key can be used to retrieve the state associated with a widget.

For more information, see the [`GlobalKey`](https://api.flutter.dev/flutter/widgets/GlobalKey-class.html)
 API.

    *   **for:** _for( ..; ..; )_
        
    *   **foreach:** _**for(.. in ..)**_

Copy

      var year = 2010;
      var myName = 'Empar';
    
      // IF
      if (year >= 2001) {
        // prints: 21st century
        print('21st century');
      } else if (year >= 1901) {
        print('20th century');
      }
      
      // SWITCH
      var fruit = 'tropical orange';
    
      switch (fruit) {    
        case 'apple':
          print('It\'s an apple!');
          // break;             // not necessary when case is not empty      
        
        case 'banana':
          print('It\'s a banana!');   
        
        case 'orange':  // empty case --> jumps next: 'mandarine'
        case 'mandarine':
          print('It\'s a type of orange!');

          print('It\'s a tropical fruit');

          print('Unknown fruit');
      }  
      
      // SWITH EXPRESSION
      String myChar = '0';
      String tipo = switch (myChar) {
        '/' || '*' || '+' || '-' => 'Es un operador',

        _ => 'No está permitido', // also use: default instead of _
      };
    
      print(myChar + ' => ' + tipo); // PRINTS: '0 => No está permitido'

      }

        print(month);
      }
    
      // prints: 2011..2015
      while (year < 2016) {
        year += 1;
        print(year);
      }
    
      // prints: 2016..2022
      do {
        print(++year);
      } while (year < 2022);
    }

# dynamic vs Object | Dart

En Dart, `**dynamic**` y `**Object**` son dos tipos que podemos usar cuando no estamos seguros del tipo de una variable. Pero tienen algunas diferencias importantes:

*   `**dynamic**` no es realmente un tipo, sino que es una forma de **desactivar** **la** **comprobación** **de** **tipos**. `**dynamic**` Acepta **cualquier** **tipo** de valor incluido _null._ Cuando declaramos una variable como dynamic, le estamos indicando al analizador estático que confíe en nosotros y que no compruebe nada, por lo que **no lanzará ningún error** en su uso.
    
*   `**Object**` es el tipo de datos 'base' de todos los tipos de datos no nulos (Object? que accepta también null). Cuando definimos una variable como _Object,_ el analizador estático **lanzará** **una** **error** si se realizan operaciones no permitidas en _Object_, por lo que se tendrá que convertir al tipo deseado antes de su uso o **no compilará**.
    

Source: [https://www.cloudhadoop.com/dart-dynamic-vs-object/](https://www.cloudhadoop.com/dart-dynamic-vs-object/)

[](https://ecm-pmdm-flutter.gitbook.io/dart/conceptos-basicos/dynamic-vs-object#ejemplos)

Ejemplos


-------------------------------------------------------------------------------------------------------

Las variables `**dynamic**` pueden ser asignadas a cualquier tipo de datos (y null) y estas variables se pueden asignar a cualquier otro tipo de variable.

Copy

    void main() {
       //dynamic variable assigned with string 
      dynamic name = "john";
    
      // you can assign dynamic variable to any other variable
      // no static type checking!
      String fname = name;      
      int number = name;   // types not match --> Execution Error  !!
    }

Cuando declaramos una variable del tipo **Object**, podemos asignarle cualquier tipo de datos, pero deberemos realizar **casting** (typecase) cuando las asignamos a otro tipo de variable.

Copy

    void main() {
      Object age = 25;
    
      int age1 = age as int;   // static type check --> typecase before assigning!
      
    }

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/conceptos-basicos/dynamic-vs-object#valores-null)

**Valores Null**

Copy

    main() {
      dynamic str = null;          // No error
      Object obj = null;           // Compile time error !!
      Object? objnullable = null;  // No error
    }

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/conceptos-basicos/dynamic-vs-object#acceso-a-metodos-y-propiedades)

Acceso a métodos y propiedades

Copy

    class Employee {
      printEmp() => print("Employee Object");
    }
    
    void main() {
      dynamic a = new Employee();
      Object b = new Employee();
      a.printEmp(); // No error
      b.printEmp(); // Compile error
    }

**Explicación del código:**

*   Cuando creas una instancia de la clase `**Employee**` con un tipo dinámico, puedes acceder a los métodos y propiedades.
    
*   En cambio, lo mismo no funciona para `**Object**`, los métodos y propiedades no son accesibles.

# Programación funcional | Dart

Dart da un buen soporte a la programación funcional, similar a Kotlin. La programación funcional tiene un papel muy importante en Dart.

En Dart todo son objetos, las funciones son también objetos y tienen un tipo: _Function_. Por tanto, podemos:

*   Asignar a una variable una función e invocarla
    
*   Pasar una función como argumento de otra función.
    

Veamos con ejemplos los conceptos de **Lambdas, Higher-Order functions** y **Closures**.

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/programacion-funcional#lambdas)

Lambdas

Una "Expresión lambda" es una función sin nombre, también se llama _**Función anónima**_ o simplemente _**Lambda.**_

Ejemplo:

Copy

    // normal functions have a name
    int addition(int x, int y) {  
      return x + y;  
    }
    
    void main() {
      
      // 1. lambda expression (function is nameless): only has parameters and body
      //--> assign it to a variable (of type Function, or use var) 
      Function addFunc = (int x, int y) {  
                            return x + y;  
                          };
      
      // 2. lambda with shorthand syntax: Fat Arrow
      var subsFunc = (int x, int y) => x - y;
    
      // use addFunc 
      print(addition(30,20));   // prints: 50
      print(addFunc(2,3));      // prints:  5
      print(subsFunc(5, 9));    // prints:  -4
    }

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/programacion-funcional#higher-order-function)

Higher-Order Function

Una función de orden superior és aquella que:

*   Acepta una función como parámetro, o que
    
*   Retorna una función, o
    
*   Las dos cosas a la vez
    

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/programacion-funcional#lexical-closures)

Lexical closures

> **Lexical scope** (ámbito léxico): Dart es un lenguaje de ámbito léxico, es decir, el ámbito de las variables se determina de forma estática, por su disposición al código, siguiendo las { }. De modo que, desde los niveles más internos se puede acceder a todo lo que hay en los niveles superiores.

Un Closure es una función especial. El closure puede acceder y modificar los valores de variables presentes en su ámbito léxico, incluso cuando se usa fuera de su ámbito original. (En Java 8, no está permitido, pero sí en otros lenguajes como Kotlin)

Ejemplo:

Copy

    // function that multiplies two numbers
    double multiply(double a, double b) => a * b;
    
    // Returns a closure (Function) that multiplies [value] to the function's argument [x]
    Function makeMultiplier(double value) {
      return (x) => multiply(value, x);   // [value] can be reached outside lexical scope
    }
    
    void main(){
      // assign function makeMultiplier to variables
      Function mult5 = makeMultiplier(5.0);   // mult5 holds value = 5.0
      Function mult10 = makeMultiplier(10.0); // mult10 holds value = 10.0
      
      // mult5 and mult10 has access to 'value' that is out of main lexical scope
      print(mult5(10));     // prints 50
      print(mult10(2));     // prints 20
    }

Tipado dinámico: _**dynamic**_

# Funciones | Dart

En Dart todo son objetos, incluidas las **funciones** que tienen su propio tipo: **Function**. Eso significa que podemos:

*   Pasar una función como parámetro a otra función,
    
*   Devolver una función desde otra función,
    
*   Asignar una función a una variable.
    

Dart también permite declarar una función dentro de otra función.

![](https://ecm-pmdm-flutter.gitbook.io/dart/~gitbook/image?url=https%3A%2F%2F1551469701-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FsGv3FSfnl3pEE6ftH9Ys%252Fuploads%252FnVY1fseIEs6KEv9WmUeJ%252Fimage.png%3Falt%3Dmedia%26token%3Da1887424-b760-40e2-be65-a1936ab3b223&width=768&dpr=4&quality=100&sign=746741d7&sv=2)

Todas las funciones (denominadas _métodos_ si están dentro de una clase) han de **devolver** un valor de un **tipo** o si no, se declaran _**void**_ y no hay _return._

*   Se recomienda siempre declarar el tipo de dato devuelto.
    
*   Si se **omite** el **tipo** de **retorno** de la función y se **omite** el return se **devuelve** por defecto _**null**__._
    
*   Las funciones que no pertenecen a ninguna clase son las llamadas _**top-level functions**__._
    

**Ejemplos**:

Copy

    void main() {
      int total = add(5, 2);
    
      print("Total sum of 5 + 2 = $total");  // prints: Total sum of 5 + 2 = 7 
    
      printHello('Empar'); // prints: Hello Empar !!
    
      print("5 * 4 = ${mult(5, 4)}");  // prints: 5 * 4 = 20
      print("-5 * 4 = ${mult(-5, 4)}");   // prints: -5 * 4 = null
    }
    
    int add(int a, int b) {
      return a + b;
    }
    
    void printHello(String name) {
      // no return type
      print('Hello $name !!');
    }
    
    mult(int a, int b) {   // return type omitted
      if (a>0 && b>0) return a * b;  
      // returns null in othercase
    }

[](https://ecm-pmdm-flutter.gitbook.io/dart/funciones#sintaxis-abreviada-de-funciones-notacion-arrow-greater-than)

**Sintaxis abreviada de funciones: notación arrow** `**=>**`


------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Si una función **sólo** **contiene** **una** **expressión** podemos utilizar una **notación** **abreviada**:

`**=> <expresion>;**`

> **Nota:** Sólo puede aparecer una expresión —no una sentencia— entre la flecha `**=>**` i el punto i coma (;). Es decir, por ejemplo, no podemos poner una sentencia `if`, pero sí una expresión condicional: `? :` .

Por ejemplo, la función `add` del ejemplo anterior se puede escribir en una sola línea:

Copy

    int add(int a, int b) => a + b;

[](https://ecm-pmdm-flutter.gitbook.io/dart/funciones#funcion-como-argumento)

**Función como argumento**


-------------------------------------------------------------------------------------------------------------

Podemos pasar una función como argumento de otra función:

Copy

    // function as argument 
    void main() {
      int a = 5;
      int b = 4;  
      
      print("$a + $b = ${calculate(a, b, add)}");  // prints: 5 + 4 = 9  
      print("$a * $b = ${calculate(a, b, mult)}"); // prints: 5 * 4 = 20
    }
    
    int add(int a, int b) => a + b;
    int mult(int a, int b) => a * b;
    
    int calculate(int a, int b, Function doOperation) => doOperation(a, b);
    
    

[](https://ecm-pmdm-flutter.gitbook.io/dart/funciones#parametros)

Parámetros


---------------------------------------------------------------------------------

En una función se pueden declarar cualquier número de **parámetros** **posicionales** **requeridos (**es **obligatorio** **especificarlos** **todos** en la llamada a la función) seguido de:

1.  **Parámetros con nombre** (_named_)
    

**o**

1.  **Parámetros** **posicionales opcionales** _(optional positional)_,
    

**Atención: NO puede haber a la vez parámetros del tipo (1) y tipo (2).**

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/funciones#parametros-con-nombre)

Parámetros con nombre

Permiten establecer el valor del **parámetro** **por** su **nombre** (muy conveniente cuando la función tiene muchos parámetros).

> En Flutter, los constructores pueden llegar a ser bastante complejos así que se recomienda utilizar sólo parámetros con nombre.

*   Se declaran en la función entre `**{ }**`**después de los parámetros posicionales requeridos**.
    
*   Cuando se llama a la función, **especificaremos** el parámetro con su nombre y su valor con la **sintaxis**: (el orden no importa) `**nombreParametro: valor**`
    
*   Por **defecto**, son _**opcionales**_ a no ser que se especifiquen como `**required**`_._
    
*   Se puede definir un **valor** por **defecto** con `**=**` (**obligatorio** si es opcional y no es _nullable_)
    

**Ejemplo:**

Copy

    void main() {
      print(add(5, d: 5));                   // prints: 15 
      print(add(5, b: 10, d: 15));           // prints: 35 
      print(add(5, d: 20, c: 10, b:100));    // prints: 135 
    }
    
    // b is optional nullable optional named parameter, 
    // c is optional named parameter with default value
    // d is required named parameter
    int add(int a, {int? b, int c = 5, required int d}) {  
      if (b == null)
        return a + c + d;
      else    
        return a + b + c + d;
    }

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/funciones#parametros-posicionales-opcionales)

Parámetros posicionales opcionales

*   Los parámetros posicionales que estén entre `**[ ]**` serán **opcionales**, se declaran **después de los parámetros posicionales requeridos.**
    
*   Se les puede asignar un **valor** por **defecto** con `**=**` (obligatorio si no es un parámetro _nullable_)
    

**Ejemplo**:

Copy

    void main() {    
      print(add(1));        // prints: 21
      print(add(1, 2));     // prints: 13
      print(add(1, 2, 3));  // prints: 6
    }
    
    int add(int x, [int? y, int z = 10]) { // y, z are optional positional parameters
      var total = x; 
      total += y ?? 10;   // if y!=null adds y otherwise adds 10 
     
      return total+=z;
    }

# Variables | Dart

Dart es un lenguaje de **tipado** **estático**, lo que significa que cada variable en Dart tiene un tipo que debe conocerse cuando se compila el código. El **tipo** de la variable **no** puede **cambiar** durante la ejecución del programa. C, Java, Swift y Kotlin también son de tipado estático.

> Esto contrasta con lenguajes como Python y JavaScript, que son de _tipado dinámico_. Eso significa que las variables pueden contener diferentes tipos de datos cuando ejecuta el programa. No necesita saber el tipo cuando compila el código.

Copy

    // Prints on Console the value of integer variable: 153
    void main() {
        int length = 153;
        print(length);
    }

Además, Dart realiza la **inferencia** **de** **tipos** cuando no se define de forma explícita el tipo de datos. Así, anteponiendo la palabra reservada `**var**` al nombre de variable, se le dice al compilador que debe determinar (inferir) el tipo de variable a partir del valor que se le asigne.

Copy

    // Prints on Console the value of infered double variable: 3.141516
    void main() {
      var pi = 3.141516;
      print(pi);    
    }

**Atención** al definir los nombres de identificadores:

*   Deben ser **únicos**.
    
*   Los identificadores **no** **pueden** **contener** **palabras** **reservadas**.
    
*   **No** están **permitidos** los **números** al **inicio** ni los caracteres en blanco o especiales, a excepción del guion bajo “\_” y el símbolo de dólar “$”.
    
*   Los identificadores **diferencian** entre **mayúsculas** y las **minúsculas**.
    

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/conceptos-basicos/variables#modificadores)

Modificadores

Las variables pueden tener los siguientes modificadores:

*   `**final**`: Una variable final se puede inicializar solo una vez, no se puede reasignar, debe inicializarse antes de utilizarse.
    
*   `**const**` - Una variable const es una constante en tiempo de compilación. Una constante es implicitamente `final`.
    
*   `**static**`: las variables estáticas están disponibles en la clase en lugar de la instancia.
    

Un _**"literal"**_ és una constante en tiempo de compilación: un número, una expresión que se evalúa a número, _null_, _true/false_ o una cadena.

#### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/conceptos-basicos/variables#ejemplo)

Ejemplo:

Copy

    void main() {  
      const String hello = "Hello World!";   // constant String
      const upperLimit = 10;  // infered integer constant
      final age;   // final variable
      
      print(hello);        // prints constant String value: Hello World
      print(upperLimit);   // prints infered constant int value: 10
      
      age = 25;      // final variable assigned to int value: 25
      // age = 30;   // reports error: as age variable is already assigned
      print('age = ' + age.toString());    //  prints 25
    }

**Atención**:

*   Si no hay intención de cambiar el valor de una variable s_e_ ha de usar el **modificador** `**final**` o `**const**`.
    
*   El valor de una variable `const` se determina en **tiempo de compilación** y **no se puede cambiar** en tiempo de ejecución (ni tampoco el valor de sus atributos ) es **inmutable**
    
*   El valor de una variable `**final**` **no se puede cambiar** en tiempo de ejecución (pero sí sus atributos, si apunta a objeto)
    

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/conceptos-basicos/variables#null-safety)

Null-Safety

*   Dart es "**null** **safe**".
    
*   Los valores _**null**_ son usualmente causa de múltiples errores durante la ejecución de los programas.
    
*   Por ello, en Dart por defecto es **obligatorio** que **todas** las variables han de estar **inicializadas** a algún valor antes de poder utilizarlas o se produce un error de **compilación**.
    

Copy

    void main() {
      String cadena;    // variable sin inicializar
      print(cadena); 
    }

El ejemplo anterior da un **Error de compilación:**

Error compiling to JavaScript: Info: Compiling with sound null safety

Warning: Interpreting this as package URI, 'package:dartpad\_sample/main.dart'. lib/main.dart:3:9:

Error: Non-nullable variable 'cadena' must be assigned before it can be used.

print(cadena);

^^^^^^ Error: Compilation failed.

*   Para **declarar** que una variable pueda tener un valor `null` se añadirá `**?**` al tipo de dato.
    

**Ejemplo:**

Copy

    main() {
      int? i;
      print(i);  // prints null
    }

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/conceptos-basicos/variables#variables-private)

Variables private

En Dart no existe el modificador _private_.

Si queremos **indicar** que una **variable** (o método) sólo se pueda utilizar en su biblioteca (cada fichero dart es una biblioteca o _library_ ) se **prefija** con **guión** **bajo**:

`int _myPrivateVariable; //visible sólo en este fichero dart`

# Concurrencia-Programación asíncrona | Dart

Dart es un lenguaje _**single-threaded.**_ Esto no quiere decir que Dart no pueda ejecutar código en paralelo.

Cuando iniciamos una aplicación Dart (con o sin Flutter), el _runtime_ de Dart la arranca dentro de un nuevo thread aislado: _**main isolate.**_ El runtime de Dart modela los threads como _**isolates**_, es decir, completamente aislados unos de otros, no comparten memoria, lo cual evita la necesidad de bloquear la memoria para evitar las _race conditions.._

Para operaciones computacionalmente largas, se pueden lanzar varios isolates y así evitar bloquear el main isolate del programa. El código dentro de cada isolate individual es single-threaded, lo cual significa que sólo una operación se ejecuta a la vez.

En el siguiente vídeo se explica cómo ejecutar código en un nuevo _isolate_: [https://www.youtube.com/watch?v=PPwJ75vqP\_s](https://www.youtube.com/watch?v=PPwJ75vqP_s)
 (2 mins)

**Ejemplo de función síncrona**: (mientras se está ejecutando y no termine, no se pueden procesar clics y otros eventos/tareas en el isolate)

Copy

    void syncFuncExample() {
      var count = 0;
      
      for (int i = 0; i < 1000000; i++) {
        count++;
      }
    }

Entonces, ¿cómo **evitamos** **bloquear** el _isolate_ actual mientras esperamos que las operaciones de cierta duración acaben? (I/O: network, BD, files,..).

Aquí es donde entran en acción los patrones de la **programación asíncrona**.

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/concurrencia-programacion-asincrona#estructura-de-un-isolate)

Estructura de un _isolate_

Dart implementa el siguiente modelo asíncrono (event-loop) de forma que el programa es capaz de manejar eventos que ocurren de forma no predecible pero sin bloquear los eventos de usuario (UI responsive)

![](https://ecm-pmdm-flutter.gitbook.io/dart/~gitbook/image?url=https%3A%2F%2F1551469701-files.gitbook.io%2F%7E%2Ffiles%2Fv0%2Fb%2Fgitbook-x-prod.appspot.com%2Fo%2Fspaces%252FsGv3FSfnl3pEE6ftH9Ys%252Fuploads%252FQT2zYppnU1XqqpIfQNm5%252Fimage.png%3Falt%3Dmedia%26token%3D2a56e9b7-67c6-4e80-90e6-0655524ae059&width=300&dpr=4&quality=100&sign=9a9a8884&sv=2)

**Event-loop**: La aplicación pasa la mayor parte del tiempo ejecutando las tareas de cada cola. Si no hay nada urgente, se ejecutan tareas como el garbage-collector.

**Microtasks**: Son tareas de corta duración con más prioridad que los eventos.

**Events**: Se añadirán eventos a la cola para que puedan ser ejecutados debido a: interacciones del usuario (clicks, teclas, gestos,..), I/O desde almacenamiento local o desde la red, Timers, Futures, Streams.

> En Flutter, se utiliza la programación asíncrona que permite mantener la UI atenta a las acciones del usuario mientras se completan las tareas de cierta duración.

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/concurrencia-programacion-asincrona#conceptos-basicos)

Conceptos básicos

*   **Operación síncrona**: operación que bloquea a otras operaciones hasta termina de ejecutarse.
    
*   **Función síncrona:** función que sólo realiza operaciones síncronas.
    
*   **Operación asíncrona:** Una vez iniciada, la operación asíncrona permite que otras operaciones se realizen antes de que ésta termine.
    
*   **Función asíncrona:** realiza, al menos, una operación asíncrona y puede realizar también otras operaciones síncronas.

# Map | Dart

Documentación oficial: [https://api.dart.dev/stable/dart-core/Map-class.html](https://api.dart.dev/stable/dart-core/Map-class.html)

Cuando necesitamos una estructura para guardar **pares** **clave-valor**, la clase **Map** es una buena elección.

Un Map es una **colección** de pares **clave-valor** donde obtenemos el valor a partir de la clave (diccionario).

Dart proporciona soporte nativo para Maps. Un Map en Dart es **similar** a los diccionarios de Swift o maps de Kotlin o **Java**.

[](https://ecm-pmdm-flutter.gitbook.io/dart/colecciones/map#caracteristicas)

**Características**


-----------------------------------------------------------------------------------------------------

*   Un Map se puede **iterar**. Cada elemento se dice que es un par clave-valor.
    
*   Tanto la **clave** como el **valor** pueden ser de **cualquier** **tipo**.
    
*   La **clave** debe ser **única**, pero el valor puede repetirse.
    
*   Para **acceder** a un valor del Map debemos proporcionar la clave. Si la clave no se encuentra devuelve un valor `**null**`.
    
*   Se puede **cambiar** el valor para una clave existente.
    
*   Se pueden **añadir** dinámicamente más elementos clave-valor.
    

[](https://ecm-pmdm-flutter.gitbook.io/dart/colecciones/map#creacion-de-maps)

**Creación de Maps**


-------------------------------------------------------------------------------------------------------

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/colecciones/map#mediante-literales)

Mediante Literales

El map se declara entre `**{ }**` y `**,**` para separar los pares clave-valor.

Copy

    // Map con clave String  y  valor int
    main() {
      Map<String, int> calories = {
        'cake': 550,
        'donuts': 145,
        'cookies': 75,
      };
      
      print(calories); // prints: {cake: 550, donuts: 145, cookies: 75}
    }

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/colecciones/map#mediante-el-constructor)

Mediante el constructor

Copy

    //Map con clave String  y  valor String
    main() {
      Map<String, String> loginInfo = Map(); // empty Map of <String, String>
     
      print(loginInfo); // prints: {}
    }

[](https://ecm-pmdm-flutter.gitbook.io/dart/colecciones/map#gestion-de-los-elementos-del-map)

**Gestión de los elementos del Map**


---------------------------------------------------------------------------------------------------------------------------------------

Utilizamos los `**[ ]**` y dentro escribimos la clave.

Copy

    main() {
      Map<String, int> loginCodes = Map();  // equivale a poner el literal: {}
     
      loginCodes['empar'] = 1234;
      loginCodes['pep'] = 4321; 
      //loginCodes['joan'] = '5467';  // error:  value must be int not a String
        
      print(loginCodes); // prints: {empar: 1234, pep: 4321}
      
      print(loginCodes["empar"]);  // prints: 1234
      print(loginCodes['joan']);   // prints: null (key 'joan' not found)
      
      
      Map<String, int> calories = {'cake': 550, 'donuts': 145, 'cookies': 75, };
      print(calories); // prints: {cake: 550, donuts: 145, cookies: 75}
      
      calories['cake'] = 334;   // value of 'cake' is updated
      
      print(calories['cake']);  // prints: 334 (instead of old value: 550)  
    }

**Propiedades**


---------------------------------------------------------------------------------------------

*   **keys**: devuelve un Iterable con las claves.
    
*   **values**: devuelve un Iterable con los valores.
    
*   **length**: devuelve el número de elementos del Map.
    
*   **otras propiedades**: isEmpty, isNotEmpty, entries, ....

# Excepciones | Dart

[https://dart.dev/language/error-handling](https://dart.dev/language/error-handling)

En Dart, podemos capturar y lanzar excepciones (_**exceptions**_).

> Una excepción es un error que indica que ha ocurrido algo inesperado.

Dart tiene una **jerarquía** de herencia de excepciones (similar a la de Java). La jerarquía permite capturar tipos de errores de manera granular, yendo de lo más específico a lo más general.

Copy

    Object
     └─ Throwable (interfaz)
         ├─ Exception (interfaz)
         │   ├─ IOException
         │   │   ├─ FileSystemException
         │   │   ├─ SocketException
         │   │   └─ HttpException
         │   ├─ FormatException
         │   ├─ TimeoutException
         │   └─ ... (y muchas otras)
         │
         └─ Error (clase)
             ├─ ArgumentError
             ├─ StateError
             ├─ TypeError
             ├─ RangeError
             └─ ... (y muchas otras)

Las `Exception` son errores esperados que deberemos manejar, y los `Error` son señales de que tu código tiene un bug que necesita ser corregido.

*   A diferencia de Java, todas las excepciones son **unchecked** y los **métodos** **no** estan **obligados** ni a **declarar** qué excepciones lanzan **ni** a **capturar** ninguna excepción.
    
*   **Si una excepción no se captura no hay oportunidad para recuperarse del error, el hilo actual de ejecución (isolate) se suspende y el programa termina anómalamente**.
    
*   En Dart **se puede lanzar cualquier tipo de objeto no-nulo como excepción**, no sólo el tipo Exception y Error:
    
    *   **Exception**: La usaremos cuando la intención es capturarla e informar al usuario para intentar recuperarse del error: FormatException, IOException,...
        
    *   **Error**: Son errores no controlados y de los que es difícil recuperarse como index-out-of-range (error en la lógica) , out-of-memory...: IndexError, OutOfMemoryError,...
        
    
*   Podemos definir nuestros propios tipos de Exception/Error.
    

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/excepciones#captura-de-excepciones-try-..-on-..-catch-..-finally)

Captura de excepciones: try .. on .. catch .. finally

Dart recomienda ser específico en el tipo de excepción que se captura (no usar solo Exception) y así poder dar información del tipo concreto de error.

Para capturar las excepciones podemos usar **solo** `**on**` o **solo** `**catch**` o **ambos** (**on** ... **catch(e))**:

*   Usaremos `**on**` `<tipo_exception>` si solo queremos especificar el **tipo** de **excepción**.
    
*   Usaremos `**catch(..)**` cuando necesitemos el objeto _exception_ para tratarla. Podemos especificar `catch` con un parámetro: `catch(e)` o con dos parámetros: `catch(e, s)`. El primero es el objeto _**exception**_ que fue lanzado y el segundo parámetro es el _**stacktrace**__._
    

Podremos capturar más de un tipo de excepción con **múltiples bloques** `**on/catch**`. El primer bloque catch que coincida con el tipo del objeto exception lanzado será el que tratará la excepción.

Si el bloque `**catch**` **no** **especifica** el tipo de excepción (on `<tipo_exception`), podrá tratar **cualquier** **excepción** (ha de ser el último).

**Ejemplo**:

Copy

    var list = [1, 2, 3];
    
    void main() {  
      print("LIST: $list\n");    // prints: LIST: [1, 2, 3]
      
      try {    
        // trying different catch exception/error examples     
        exampleFormatException('3,14');  // throws: FormatException, 3,14 is not a double number
        //exampleRangeError(3);          // throws: RangeError: RangeError (index): Index out of range: index should be less than 3: 3
        //exampleUnsuportedOperationError(0);   // throws: Error, Unsupported operation: Result of truncating division is Infinity: 2 ~/ 0
          
      } on FormatException {  // catch(e) ommited (we don't need exception object)
        print('FormatException, 3,14 is not a double number');
      
      } on RangeError catch (e) {  // e is the error object thrown
        print('RangeError: $e');
    
      } catch (e, s) {    // Handle general error in the last place, s is StackTrace object
        print('ERROR: $e');   
        print('\nSTACK TRACE: $s');
        
      } finally {   // always executes
        list.clear();    
      }
      
      print("\nLIST CLEARED: $list");   // prints: LIST CLEARED: []
    }
    
    void exampleFormatException(var number) {
      print(double.parse(number));  
    }
    
    void exampleUnsuportedOperationError(int divider) {
      print(list[1]~/divider);  
    }
    
    void exampleRangeError(int index) {
      list[index] = 10;   
      print(list[3]);
    }

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/excepciones#throw)

Throw

*   Permite **lanzar** excepciones.
    
*   Se puede lanzar cualquier objeto, no sólo Exception y Error.
    

Ejemplo:

Copy

      try {
        if (3 > list.length - 1) {
          throw RangeError("Attention->index 3 out of list");
        }
      } catch (e) { // general exception
        print('$e'); // prints: RangeError: Attention->index 3 out of list
      }
      
      try {
        if (list[2] > 2) {
          throw "${list[2]} must be a value <=2";   // error object is of type String
        }   
      } catch (e) {  // general exception
        print('$e'); // prints:  3 must be a value <=2
      }
    }

[](https://ecm-pmdm-flutter.gitbook.io/dart/excepciones#custom-exception)

      }

# Clases | Dart

Las **clases** en **Dart** son **similares** a **Java**_**.**_ De herencia simple, **todos** los **objetos** en Dart, incluso los tipos primitivos, son instancia de una clase.

Todas las clases, excepto Null, derivan de la clase Object que proporciona por defecto:

*   toString() -> devuelve la representación en cadena del objeto
    
*   hashCode -> _getter_ que devuelve el código hash del objeto
    
*   operador `**==**` \-> permite determinar si dos objetos son iguales
    

**A tener en cuenta en Dart:**

*   El uso de **new** para instanciar objetos es **opcional (no se escribe por convención)**
    
*   No existen los calificadores: _**public, private o protected**__._
    
*   Todos los **miembros** de la clase son **públicos** **por** **defecto**.
    
*   Para hacer un **miembro** **privado** en la _library_ (toda aplicación Dart es por defecto una _library_) usaremos la **barra baja \_** como prefijo al nombre del miembro.
    
*   Cada variable (atributo) de instancia tiene implícito un **getter** y un **setter**. También podemos implementarle las propiedades `get` y `set` si es necesario.
    
*   No se puede definir una clase dentro de otra clase (en Java sí podemos anidar clases).

# Tipos enumerados | Dart

Los tipos enumerados o `enum` son una clase especial de Dart para **representar** un **número** **fijo** de **valores** **constantes.**

Cada valor en un `enum` tiene una propiedad `getter` llamada `index`, que devuelve la **posición** del valor en el enum comenzando por el índice 0, como en las listas. Así, el primer valor tiene el índice 0, el segundo valor tiene el índice 1,....

**Ejemplo**:

Copy

    enum Color { red, green, blue }
    
    void main() {
      List<Color> myColors = Color.values; // get all colors
    
      // Print: all values of enum: [Color.red, Color.green, Color.blue]
      print(myColors);
    
      var aColor = Color.blue;
    
      // Print: Color.blue --> has index: 2
      print('Color.${aColor.name} --> has index: ${aColor.index}');
    
      switch (aColor) {
        case Color.red:
          print('Red as roses!');
          break;
        case Color.green:
          print('Green as grass!');
          break;
        default: // Without case default:, you'll see a WARNING.
          // prints: Sorry, blue not allowed
          print("Sorry, ${aColor.name} not allowed"); 
      }
    }

# Futures, async/ await | Dart

Dart soporta la programación asíncrona utilizando objetos del tipo **Future,** Streams y expresiones **async**/**await**

Un Future es el tipo de objeto que devuelve una operación asíncrona. El _future_ representa el resultado de esa operación asíncrona.

> Un Future es una promesa de que el objeto tendrá en algun momento del futuro un valor (o un error). Es similar al Promise de Javascript.

El objeto _future_, puede estar:

*   _**incompleto:**_ todavía sin valor porque la operación está en la cola para ser ejecutada (o no ha terminado)
    
*   _**completo:**_ la operación ya ha terminado y el valor ya está disponible o termina con un error.
    

> Hasta que el Future no está completo, su valor, no se puede utilizar.

Para realizar operaciones asíncronas tenemos dos opciones:

*   Podemos utilizar directamente el API de Future y Stream
    
*   Utilizar _async_ y _await_ que es lo que se recomienda: el código asíncrono queda más claro al parecerse más a la forma en que escribimos el código síncrono.
    

Podemos **crear** una **tarea** **asíncrona,** y que se añada a la cola de eventos, con uno de los constructores de la clase **Future**, por ejemplo:

*   Future( <función a ejecutar> ) o,
    
*   Future.delayed( <tiempo>, <funcion a ejecutar cuando termine el tiempo>)
    

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/concurrencia-programacion-asincrona/futures-async-await#ejemplo-future-api)

Ejemplo: Future API

En el siguiente ejemplo vamos a simular que descargamos un fichero. La tarea tiene una duración de unos 6 segundos. Esta tarea se realizará de forma asíncrona para no bloquear el main(), se devolverá el valor del objeto Future utilizando el método Future.then() o el Error:

Future completes with Value

Future completes with Error

Copy

    Future<String> downloadFile() {
      Future<String> result = Future.delayed(Duration(seconds: 6),
                () => "Cats, dogs and other pets");  // lambda is called when time expires -->
                                                     // that returns a String (that simulates file content)    
        
      return result;  // returns Future<String>: future object that will contain a String value
    }
    
    void main() {
      print('main starts !');  
      
      // calling downloadFile() function does not block main, 
      // returns immediately (with incomplete future)
      
      // can't print the value because value is not yet available
      // we get printed: Instance of '_Future<String>' 
      print(downloadFile());  
      
      // to get the value: 
      // use .then with the callback to execute if future completed with value
      downloadFile().then( (s) => print(s)); // s is the return value of Future<String>
        
      print('main finishes !');      
    }

El programa anterior muestra:

> main starts ! Instance of '\_Future' main finishes ! Cats, dogs and other pets

    Future<String> downloadFile() {

    void main() {
      print('main starts !');  
      
      downloadFile().then((s) => print(s))  // callback if future completes with a String value: s
                    .catchError((e) => print(e));  // e is error message

      // downloadFile().then((s) => print(s), onError: (e) => print(e));

### 

[](https://ecm-pmdm-flutter.gitbook.io/dart/concurrencia-programacion-asincrona/futures-async-await#ejemplo-async-await)

Ejemplo: async/await

Dart proporciona los modificadores _**async**_ y _**await**_ de la biblioteca _**dart:async**_, esto nos permitirá escribir código asíncrono como si fuese sincrono.

Si queremos que una función pueda ejecutar código de forma asíncrona, deberemos de indicarlo con el modificador _**async**_ después del nombre de la función.

> **Una función** _**async**_ **siempre devuelve un objeto Future**

Si queremos esperar a que una operación asíncrona acabe, usaremos el modificador _**await**_. Cuando una función utiliza en su código _**await**_, deberemos **necesariamente** marcarla como _**async**_.

> 🖐️ATENCIÓN: Una función _**async**_ ejecuta su código **síncronamente** hasta que encuentra el primer _**await**_, entonces retorna el objeto Future y la operación a realizar se pone en cola para ejecutarse asíncronamente. La función reanudará su ejecución cuando el await termine.

Veamos el ejemplo anterior utilizando _async_ i _await:_

Copy

    // downloadFile() is now asynchronus 
    // marked with "async" modifier: async function always returns Future object
    Future<void> downloadFile() async {  
      
      // Careful! --> function executes "synchronously" until it finds first await expression:   
      print('...getting file from server...'); 
        
      // at this point, function returns immediately a Future object
      // await operation is queued to execute asynchronously  
      String fileContent = await Future.delayed(Duration(seconds: 6),
                                                () => " Cats, dogs and other pets");  // lambda is called when time expires --> returns String that simulates file content
      
      // async funcion resumes when await expression completes: fileContent expression is ready
      print("\nTHE CONTENT OF FILE IS: $fileContent");	
    }
    
    void main()  {  
      
      print('main() starts !');    // sync function
      print("async - Starting Downloading function ....");   // sync function
      
      downloadFile();   // async function: do not block main()
           
      print('main() finishes !');     
    }

El programa anterior muestra:

> main() starts ! async - Starting Downloading function .... ...getting file from server...
> 
> main() finishes !
> 
> THE CONTENT OF FILE IS: Cats, dogs and other pets

# Interfaces implícitas | Dart

*   En Dart, **no existe una sintaxis específica para declarar una interface pura** (como ocurre en Java)
    

Si la intención de una clase es ser solo una interfaz y no se quiere que se pueda hacer `extends` desde otras clases fuera de su propia biblioteca, podemos añadirle a `class` el modificador `interface` con lo cual la clase podrá implementarse pero no extenderse desde otras bibliotecas que no sean la propia.

Más información sobre **modificadores** de **clase** en: [https://dart.dev/language/class-modifiers](https://dart.dev/language/class-modifiers)

*   Las clases se usan como **interfaces** **implícitas**: Todas las clases en Dart definen de forma implícita una interfaz que contiene todos los miembros de instancia de la clase así como todas las interfaces que implementa.
    
*   Se usará cuando no queremos heredar la implementación (código) sino sólo la API de otra clase, por tanto, se han de **redefinir** **todos** los **métodos** de las interfaces que se implementen.
    
*   Una clase puede implementar una o varias interfaces (separadas por comas) declarándolas con `implements` e implementando todas las API de las interfaces implementadas.
    

[](https://ecm-pmdm-flutter.gitbook.io/dart/clases/interfaces-implicitas#ejemplo)

Ejemplo:


-----------------------------------------------------------------------------------------------

Copy

    void main() {
      Person empar = Person('Empar');
      Impostor impostor = Impostor()
        .._name = 'Impostor'; // It will have no effect on impostor's name
    
      // prints: PERSON SAYS: Hello, Anna. I am Empar.
      print('PERSON SAYS: ${getGreet(empar, 'Anna')}');
    
      // prints:  IMPOSTOR SAYS: Hi Aitana. Do you know what my real name is?--> [???]
      print('IMPOSTOR SAYS: ${getGreet(impostor, 'Aitana')}');
    }
    
    String getGreet(Person person, String toOther) => person.greet(toOther);
    
    // A person. The implicit public interface contains greet()
    class Person {
      // In the interface, but visible only in this library.
      final String _name;
    
      // Not in the interface, since this is a constructor.
      Person(this._name);
    
      // In the interface.
      String greet(String who) => 'Hello, $who. I am $_name.';
    }
    
    // An implementation of the Person interface.
    class Impostor implements Person {
      static const _impostorName = '???';
    
      @override
      String get _name => _impostorName; // ??? It's the name for any Impostor person
    
      @override
      set _name(String value) {} // do nothing for Impostor name
    
      @override
      String greet(String who) => 'Hi $who. Do you know what my real name is?--> [${_name}]';
    }

# Clases Callable (Callable objects) | Dart

*   Una _**Callable**_ _**class**_ permite **invocar** la **instancia** de la clase **como** **una** **función**.
    
*   Son esenciales en la filosofia funcional de Dart.
    
*   Para ello, la clase tiene que implementar el método `call()`
    

Puntos clave de las clases _callable_[](https://ecm-pmdm-flutter.gitbook.io/dart/clases/clases-callable-callable-objects#puntos-clave-de-las-clases-callable)

1.  **Sintaxis más limpia:** Permiten que el código sea más claro y elegante, especialmente cuando se usan objetos en contextos funcionales o en librerías que esperan funciones.
    
2.  **Facilitan la reutilización y abstracción**: Una clase callable encapsula tanto el estado como el comportamiento de forma más compacta. Así, puedes tener un objeto que contenga datos y que, al invocarlo, ejecute directamente una lógica sobre esos datos.
    
3.  **Patrones de diseño más flexibles:** Permiten imitar ciertos patrones de diseño como el patrón command, donde puedes crear objetos que representan acciones invocables.
    

**Ejemplo**

En el siguiente ejemplo, Person es una clase callable de forma que la propia instancia anna puede usarse como una función, obteniendo un **código** más **natural** y **flexible**.

Copy

    class Person {
      // makes this class a callable class
      String call(int age, String name) =>
         "Hi $name you are $age years old, so young!";
    }
    
    void main() {
      var anna = Person();
      
      // using person instance as function:
      print(anna(25, "Anna")); // prints: Hi Anna you are 25 years old, so young!
    }

# Mixins | Dart

Los **Mixins** son la forma de **reutilizar** **código** en varias jerarquías de clases.

*   Para **definir** un _mixin_, utilizamos `**mixin**` en la declaración en vez de `**class**`
    
    *   Si queremos utilizar el `mixin` como una clase normal, es decir, **vamos a instanciarlo**, lo declararemos como `**mixin class**`.
        
    *   Tanto los _**mixins**_ como _**mixin classes**_ no pueden tener **ni** _**extends**_**, ni** _**with**_**,** **ni** **constructores**.
        
    *   Para **restringir** qué clases pueden usar el mixin escribiremos después del nombre del mixin: `**on**` <nombre de la clase>
        
    
*   Para **utilizar** los _mixins_ en otras clases: escribir (si es el caso, después de _extends_) `**with**` seguido de los nombres de los mixins separados por comas.

# Constructores | Dart

*   Permiten instanciar objetos de la clase y iniciar sus miembros.
    
*   Usaremos `_this_` para referirnos a la instancia actual.
    

En Dart se recomienda usar `_this_` sólo cuando hay un conflicto de nombre.

*   Si no declaramos un constructor, la clase tiene implícito un **constructor** **por** **defecto** (mismo nombre de la clase y sin argumentos)
    
*   El **constructor** **con** **parámetros** (con mismo nombre de la clase) lo utilizaremos para iniciar las variables de instancia. Como la **inicialización** se realiza de forma frecuente, Dart permite hacerlo de una forma más simple y compacta, de forma **abreviada**:
    

Inicialización abreviada

Inicialización tradicional

Copy

    class Person {
      String? name;  // default value is null
      int? age;      // default value is null
      
      // setting name and age before body runs  
      Person(String this.name, int this.age){
        print(name);
        print(age);
      }

Copy

    class Person {
      String? name;  // default value is null
      int? age;      // default value is null
        
      Person(String name, int age) {
        this.name = name;
        this.age = age;
      }

*   En Dart, **no** se permite la **sobrecarga** de constructores, por tanto, se define:
    
    *   **Un constructor por defecto,** o
        
    *   **Un constructor con parámetros,**
        
    
    pero no los ambos a la vez.
    
*   Para tener más de un constructor, Dart permite definir **constructores con nombre** con la sintaxis: `NombreClase.nombreConstructor(parametros)`
    
*   Para llamar a un constructor de la superclase usaremos: **super**

Constructores Constantes

Constructores Factory

Si vamos a **instanciar** **objetos** q**ue no van a cambiar nunca**, podemos hacer que estos objetos sean constantes en tiempo de compilación (sólo tendremos una instancia).

Para ello, declaramos el constructor como `const` y todas las variables de instancia las declaramos como `final`.

En la llamada al constructor, se deberá anteponer `const` para indicar que instanciamos un objeto constante. Si no se llama al constructor con `const` se crearà una nueva instancia.

**Ejemplo**:

Copy

    class InmutablePoint {
      final double x;
      final double y;
    
      const InmutablePoint(this.x, this.y);
    }
    
    void main() {
      var p1 = const InmutablePoint(10, 20);  // create a constant-object
      var p2 = const InmutablePoint(10, 20);  // create a constant-object
      
      if (p1 == p2) {
        print('p1 and p2 are the same instance');
      }
      
      var p3 = InmutablePoint(10, 20);   // new instance, not constant-object
      
      if (p3 != p1 && p3 !=p2) {
        print('p3 is different from p1 and p2');
      }
    }

Los constructores que se declaran `factory` **no siempre crean una nueva instancia** de la clase, sino que **permiten** **devolver** una **instancia** **previa** (cached) o una **instancia** de un **subtipo**.

También permiten iniciar variables _final_ cuando, por la lógica asociada, no se puede hacer desde la lista de inicialización.

      logger3.log("clear()");

    class Logger {  
      // _cache is library-private, thanks to the _ in front of its name.
      static final Map<String, Logger> _cache = <String, Logger>{};
      final List<String> logMessages = [];

      bool mute = false;

        return Logger._cache.putIfAbsent(name, () => Logger._internal(name));
      }
    
      factory Logger.fromJson(Map<String, Object> json) {
        return Logger(json['name'].toString());
      }
      
      void log(String msg) {
        if (!mute) print(msg);
        // add msg to log list
        logMessages.add(msg);    
      }
      
      
    }

Ejercicio 5: Método de extensión para _int_[](https://ecm-pmdm-flutter.gitbook.io/dart/ejercicios/clases#ejercicio-5-metodo-de-extension-para-int)

Crea una extensión llamada `**RangeList**` sobre la clase _int_ que defina un método llamado `_generateRangeTo_` _:_

*   Este método tendrá como argumento un número entero: `int upTo`y devolverá una lista con todos los valores enteros dentro del rango en order creciente, incluido `upTo`.
    
*   Si el argumento `upTo` es menor que el valor actual, el método retornará una lista vacía.
    

**Ejemplo** de uso:

Copy

    void main() {
      var myRange10 = 1.generateRangeTo(10);
      print(myRange10);
      
      for (var i in 1.generateRangeTo(5)) {
        print(i);
      }
     
      print(5.generateRangeTo(15));
      print(4.generateRangeTo(4));
      print(5.generateRangeTo(0));
    }

    1
    2
    3
    4
    5

    []
    []

**Mejora:** Si el argumento es menor que el número actual, se generará la lista en orden inverso. Si son iguales, seguirá devolviendo una lista vacía.
